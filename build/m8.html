Module 8

Classical Algorithms Toolbox

01 · before any theory

You already run kNN every day

New restaurant. You've never eaten there. Is it any good? You don't run a regression. You check the three most similar places you have tried — same neighbourhood, same price, same cuisine — and you guess from those.

That's it. That's the whole algorithm called k-Nearest Neighbours. You judge a new thing by the things nearest to it. No formula, no training, no theory of the world — just "who are your closest neighbours, and what are they?"

So here's the only real decision you ever have to make with it. Don't scroll — commit to an answer first.

You ask k neighbours and take their majority vote. You're deciding between asking 1 neighbour or 500. What's the catch with each extreme?

Small k = high variance: one outlier next to your query flips the whole prediction. Huge k = high bias: you average in points from across the entire dataset, so every query drifts toward "whatever class is most common overall." The art is the middle. You'll feel this exact trade-off in the lab in about thirty seconds — it's the same bias-variance tug-of-war from Module 6, now something you can drag with a slider.

Why does the same point flip from A to B?

At k=1 the prediction is decided by the single closest training point — if that's a stray A, you predict A. At k=15 you poll fifteen points; the lone A is badly outnumbered by all the B's that surround the region, so the vote swings to B. Same query, same data — only the number of voters changed. That one observation is the whole reason k is a hyperparameter you have to tune, not a constant you can ignore.

03 · now the math, because you earned it

"Nearest" needs a ruler

You've been eyeballing "closest." The computer can't eyeball. It needs a number for distance. The default is Euclidean distance — plain straight-line distance, the Pythagorean theorem stretched to as many features as you like.

\[ d(\mathbf{a},\mathbf{b}) = \sqrt{\sum_{j=1}^{m}(a_j - b_j)^2} \]

a, b two points, each a list of feature values. m how many features. a_j, b_j feature \(j\) of each point. Plain English: for every feature, subtract, square (kills the sign), sum, square-root.

To classify a new point: measure its distance to every stored point, keep the \(k\) smallest, let those \(k\) labels vote, most common wins. There is nothing to "fit" — kNN just memorises the data. That's why it's a lazy learner: zero work at training time, all the work at prediction time.

Your data has age in years (0–100) and salary in dollars (0–200000). You run kNN with no preprocessing. What actually decides "nearest"?

Distance squares each difference. A salary gap of 10,000 contributes \(10{,}000^2 = 10^8\); the largest possible age gap of 100 contributes only \(10{,}000\). Salary wins by a factor of ten thousand. The age column might as well not exist. Scaling is not optional for kNN — standardize features first (Module 7) or distance is measured almost entirely along whichever column happens to carry the biggest raw numbers.

05 · build it in raw NumPy

kNN from scratch — predict the print, then run it

Here's the whole algorithm in pure NumPy. No library magic. You already computed the answer by hand above, so predict every line of output before you run it.

import numpy as np from collections import Counter def knn_predict(X_train, y_train, query, k=3): # Euclidean distance from the query to every training point. # (X_train - query) broadcasts the query across all rows. diffs = X_train - query # shape (n_points, n_features) dists = np.sqrt(np.sum(diffs**2, axis=1)) # one distance per point # Indices of the k smallest distances. nearest = np.argsort(dists)[:k] nearest_labels = y_train[nearest] # Majority vote: most_common returns [(label, count)], take the label. winner = Counter(nearest_labels).most_common(1)[0][0] return winner, dists, nearest X_train = np.array([[1,2],[2,3],[3,5],[5,4],[4,2]], dtype=float) y_train = np.array(["A","A","B","B","B"]) query = np.array([3,3], dtype=float) pred, dists, nearest = knn_predict(X_train, y_train, query, k=3) print("distances:", np.round(dists, 2)) print("3 nearest indices:", nearest, "-> labels:", y_train[nearest]) print("prediction:", pred)
The output

Distances [2.24 1. 2. 2.24 1.41], three nearest indices [1 4 2] (P2, P5, P3) with labels ['A' 'B' 'B'], and prediction: B — matching your hand calculation exactly. The entire "training" was storing two arrays. All the compute happens at prediction time. That's the lazy-learner tax: cheap to train, expensive to predict.

06 · the algorithm that asks questions

Decision trees: 20 questions, automated

A good 20-questions player never opens with "Is it a platypus?" They ask "Is it alive?" — a question that splits the universe in half. A decision tree is a flowchart of yes/no questions the algorithm builds for itself: at every step it searches every feature and threshold for the single split that best separates the classes, then repeats on each branch. The doctor's instinct — most informative question first — turned into code.

But "best split" needs a number. How messy is a group of examples? That's Gini impurity.

\[ G = 1 - \sum_{c=1}^{C} p_c^2 \]

C number of classes. p_c fraction of the node that is class \(c\). Plain English: square each class's share, add them up, subtract from 1.

A pure node (all one class) has \(p_c=1\), so \(G = 1-1 = 0\) — spotless. A 50/50 two-class node has \(G = 1-(0.5^2+0.5^2) = 0.5\) — maximum mess. Gini runs from 0 (pure) up to \(1 - 1/C\).

A node has 9 positives and 1 negative. Without computing exactly — is its Gini closer to 0 or to 0.5?

\(G = 1-(0.9^2+0.1^2) = 1-(0.81+0.01) = \mathbf{0.18}\). Impurity is about how lopsided the mix is, not whether two classes merely exist. 9-to-1 is very lopsided, so it's far purer than a 4-to-6 node (which sits at 0.48). The closer to all-one-class, the closer to 0.

To score a candidate split, compute the weighted impurity after the split — each child's Gini, weighted by how many examples land in it:

\[ G_{\text{split}} = \frac{n_L}{n}G_L + \frac{n_R}{n}G_R \]

n examples at the parent. n_L, n_R counts going left / right. G_L, G_R the children's Gini. Information gain \(= G_{\text{parent}} - G_{\text{split}}\): how much mess the split cleaned up.

The tree tries every feature and every threshold and keeps the split with the largest gain. (Some trees use entropy instead of Gini; they behave almost identically — we stick with Gini.)

Gini of a split, by hand

Parent: 4 positives, 6 negatives → \(G_{\text{parent}} = 0.48\) (you just saw why). A candidate split sends 4 left (3 pos, 1 neg) and 6 right (1 pos, 5 neg). Work it through:

  • Left: \( G_L = 1 - ((3/4)^2 + (1/4)^2) = 1 - (0.5625 + 0.0625) = 0.375 \)
  • Right: \( G_R = 1 - ((1/6)^2 + (5/6)^2) = 1 - (0.0278 + 0.6944) = 0.2778 \)
\[ G_{\text{split}} = \tfrac{4}{10}(0.375) + \tfrac{6}{10}(0.2778) = 0.15 + 0.1667 = 0.3167 \]

Information gain \(= 0.48 - 0.3167 = \mathbf{0.1633}\). The split dropped impurity from 0.48 to ~0.317 — a solid gain. The tree takes it.

Lock this in. "Fit a tree to the errors" comes back in two screens as boosting — and the \(\eta\) that controls each tree's correction is literally the learning rate from Module 4. Same idea, bigger steps.
07 · the rest of the cast

Four more personalities

Each algorithm below has a completely different theory of the world. Meet them fast, as characters.

Random forest — "ask a crowd, not one expert"

A single deep tree is a brilliant but erratic expert: it memorises quirks and overfits. A random forest grows hundreds of trees, each on a bootstrap sample (a same-size dataset drawn with replacement, so rows repeat and some are left out) and each split only allowed to consider a random subset of features (often \(\sqrt{m}\)). No tree is great, but their mistakes are uncorrelated, so the votes cancel the noise. Averaging \(T\) trees shrinks the variance of the random error by roughly a factor of \(T\) while leaving the real signal intact. Training on bootstrap samples like this is called bagging (bootstrap aggregating).

Gradient boosting — "a team that fixes each other's mistakes"

Boosting also builds many trees, but sequentially, each one trained to correct what the previous trees got wrong. Start with a constant guess \(F_0\). Each round, fit a small tree \(h_t\) to the current errors and add it in:

\[ F_t(\mathbf{x}) = F_{t-1}(\mathbf{x}) + \eta\, h_t(\mathbf{x}) \]

F_t the model after \(t\) rounds. h_t the new little tree, fit to the errors. η (eta) the learning rate — a small number (~0.1) capping each tree's correction. Small \(\eta\) + many trees = careful and accurate.

This is the gradient-descent idea from Module 4 — except each "step downhill" is taken by adding a whole tree instead of nudging a weight. XGBoost, LightGBM, and CatBoost are the optimized, regularized versions. On tabular data, this is usually the single strongest tool you own.

Support vector machine — "find the widest safe road"

An SVM draws a boundary \(\mathbf{w}\cdot\mathbf{x} + b = 0\) — but not just any boundary, the one with the largest possible margin, the widest empty buffer to the nearest points of either class. Wide buffer = confident, robust decision. The points touching the margin are the support vectors; they alone define the boundary, everything else is irrelevant. The kernel trick swaps the dot product \(\mathbf{x}_i\cdot\mathbf{x}_j\) for a kernel \(K(\mathbf{x}_i,\mathbf{x}_j)\) that measures similarity as if the points had been lifted into a much higher-dimensional space — so a straight margin up there shows up as a curved boundary down here, with no high-dimensional coordinates ever computed.

Naive Bayes — "multiply the clues together"

Bayes' rule flips a conditional: \(P(\text{class}\mid\text{features}) \propto P(\text{class}) \times P(\text{features}\mid\text{class})\). The "naive" move is pretending the features are independent, so the joint likelihood factors into a clean product:

\[ P(y\mid x_1,\dots,x_m) \;\propto\; P(y)\prod_{j=1}^{m} P(x_j\mid y) \]

y the class. x_j feature \(j\). P(y) the class's overall prevalence (prior). P(x_j|y) how likely that feature value is within the class. ∏ multiply over all features. Compute for each class, pick the largest.

Because it models how the data is generated within each class, naive Bayes is a generative model. Logistic regression (Module 5) models the boundary \(P(\text{class}\mid\text{features})\) directly — it's discriminative. Generative models can train on less data and even invent new samples; discriminative ones usually win on raw accuracy when data is plentiful. Most of deep learning is discriminative — except modern generative AI, which brings the generative idea roaring back at scale (Module 14).

08 · the real payoff

Six algorithms, one interface

The point of this whole module: once you understand each idea, scikit-learn lets you audition the entire toolbox by swapping one line. Six classical models on the same breast-cancer dataset. Note the StandardScaler on kNN and SVM — both are distance-based and need scaled features (you proved why above).

from sklearn.datasets import load_breast_cancer from sklearn.model_selection import train_test_split from sklearn.preprocessing import StandardScaler from sklearn.pipeline import make_pipeline from sklearn.neighbors import KNeighborsClassifier from sklearn.tree import DecisionTreeClassifier from sklearn.ensemble import RandomForestClassifier, GradientBoostingClassifier from sklearn.svm import SVC from sklearn.naive_bayes import GaussianNB X, y = load_breast_cancer(return_X_y=True) X_tr, X_te, y_tr, y_te = train_test_split(X, y, test_size=0.25, random_state=0) models = { "kNN (k=5)": make_pipeline(StandardScaler(), KNeighborsClassifier(n_neighbors=5)), "Decision Tree": DecisionTreeClassifier(max_depth=4, random_state=0), "Random Forest": RandomForestClassifier(n_estimators=200, random_state=0), "Gradient Boosting":GradientBoostingClassifier(random_state=0), "SVM (RBF)": make_pipeline(StandardScaler(), SVC(kernel="rbf")), "Naive Bayes": GaussianNB(), } for name, model in models.items(): model.fit(X_tr, y_tr) acc = model.score(X_te, y_te) print(f"{name:18s} test accuracy = {acc:.3f}")

Identical API for every model: fit to learn, score to evaluate. All six land in the mid-0.90s, with the tree ensembles and the scaled SVM typically on top and the lone decision tree slightly behind. That uniform fit/predict/score interface is scikit-learn's superpower — a whole toolbox in a for loop.

Debug it: the kNN entry was sabotaged

A teammate edited the dict. It runs without errors, but kNN's accuracy quietly cratered into the low 0.80s while every other model stayed fine. The bug is on one line. Find it before you open the fix.

models = { "kNN (k=5)": KNeighborsClassifier(n_neighbors=5), # <-- something is wrong on this line "Decision Tree": DecisionTreeClassifier(max_depth=4, random_state=0), "Random Forest": RandomForestClassifier(n_estimators=200, random_state=0), "Gradient Boosting":GradientBoostingClassifier(random_state=0), "SVM (RBF)": make_pipeline(StandardScaler(), SVC(kernel="rbf")), "Naive Bayes": GaussianNB(), }
Show the bug & why

The StandardScaler is gone — kNN is fed raw, unscaled features. The breast-cancer columns range from tiny (smoothness ~0.1) to large (area ~2500), so distance is dominated by the few big-magnitude columns and kNN measures "nearest" almost entirely along them. The fix is the exact pipeline you saw above: make_pipeline(StandardScaler(), KNeighborsClassifier(n_neighbors=5)). Same trap as the salary-vs-age predict card — distance-based models must be scaled. (SVM would suffer the same way without its scaler.)

Production-grade boosting: XGBoost

Same shape of code, the workhorse of tabular ML:

# pip install xgboost from xgboost import XGBClassifier xgb = XGBClassifier(n_estimators=300, learning_rate=0.1, max_depth=3, subsample=0.8, eval_metric="logloss", random_state=0) xgb.fit(X_tr, y_tr) print("XGBoost test accuracy =", round(xgb.score(X_te, y_te), 3))

The three key knobs: n_estimators (how many trees), learning_rate (how big each correction is — smaller needs more trees), and max_depth (how complex each tree is). These control the same bias-variance trade-off you met in Module 6.

09 · the decision you'll actually make at work

Which algorithm when?

Here's the punchline working data scientists know in their bones: on ordinary tabular data (rows and columns in a spreadsheet), a gradient-boosted tree like XGBoost still routinely beats a neural network. The fancy deep learning of Modules 10–14 rules images and text; this classical toolbox quietly wins the Kaggle competitions and powers the dashboards. A practical decision order for a new spreadsheet problem:

  • Start with a baseline: logistic/linear regression (Modules 3, 5). Fast, interpretable, tells you if the problem is even learnable.
  • Best accuracy with little fuss? Gradient boosting (XGBoost/LightGBM). Handles mixed scales, missing values, and nonlinear interactions; usually wins on tabular data.
  • Robust, low-tuning ensemble? Random forest. Great defaults, hard to break, gives feature importances for free.
  • A model a human can read aloud? A shallow decision tree — every prediction is an explainable chain of questions.
  • Small dataset, clear margins, well-scaled features? SVM can shine, especially with an RBF kernel.
  • Text/spam, very high-dimensional sparse counts, need it fast? Naive Bayes — a strong, cheap baseline.
  • Few features, smooth boundaries, zero training time wanted? kNN — but scale first, and beware large datasets (prediction is slow).

5 million rows, 80 mixed numeric + categorical features, lots of missing values, you need strong accuracy without much hand-tuning. First reach — and the one to specifically avoid?

Gradient boosting (XGBoost/LightGBM) eats this for breakfast: mixed feature scales need no scaling, missing values are handled natively, nonlinear interactions are captured automatically, top-tier accuracy with modest tuning. Avoid kNN: with 5 million stored rows it must compute distances to (potentially) all of them at prediction time — painfully slow — plus it degrades in high dimensions and needs careful scaling and categorical encoding first. A single tree would be readable but weaker; naive Bayes is fast but its independence assumption is shaky across 80 correlated features.

Cold recall — no scrolling back

Re-reading feels like learning but mostly isn't. Retrieving is. Answer each in your head first, then flip. If you can't, that's your signal to scroll up — but try first.

In kNN, what does tiny k vs huge k do to the decision boundary?
tap to flip
Tiny k = jagged, noisy boundary that overfits (high variance). Huge k = smooth, lazy boundary that blurs toward the overall majority (high bias). Tune k in between.
Why must you scale features before kNN or SVM?
tap to flip
They measure distance, which squares each feature's difference. A big-unit column (salary) swamps a small-unit one (age) and hijacks "nearest." Standardize first.
Gini impurity: what value for a pure node? For a 50/50 two-class node?
tap to flip
Pure node: G = 0. 50/50 node: G = 1−(0.5²+0.5²) = 0.5, the max for two classes. Lower = cleaner. Trees pick the split with the biggest drop in weighted Gini.
Bagging vs boosting — what's the difference and what does each reduce?
tap to flip
Bagging (random forest): independent trees in parallel, averaged → cuts variance. Boosting (XGBoost): dependent trees in sequence, each fixing the last → cuts bias, but overfits if over-trained.
Generative vs discriminative — which is naive Bayes, which is logistic regression?
tap to flip
Naive Bayes is generative (models how data is generated per class, P(features|class)). Logistic regression is discriminative (models the boundary P(class|features) directly).
First algorithm for messy tabular data with strong accuracy needs?
tap to flip
Gradient boosting (XGBoost/LightGBM). Handles mixed scales, missing values, nonlinear interactions; usually the strongest single tool on tabular data.
