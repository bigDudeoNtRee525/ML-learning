Module 6

Model Evaluation: Overfitting, Bias-Variance, Cross-Validation, Regularization

Your model just scored 99.5% accuracy. You ship it. It catches zero criminals.

Here's the trap. You built a fraud detector. Fraud is 0.5% of all transactions. A model that does literally nothing — that screams "not fraud!" at every single transaction — also scores 99.5%. Same number. One of you catches crooks; the other is a return False wearing a lab coat.

This module is one long argument that a single number can lie to you, and a toolkit for catching it in the act: how to tell whether a model learned or just memorized, how to measure the thing you actually care about, and how to tune a model so it survives data it has never seen. Don't skip the prediction cards — the whole point is to commit to a guess before you see the answer. The gap is where it sticks.

01 · the cheater's tell

A model that aces the practice exam and bombs the real one

Every model has exactly one job: do well on data it has never seen before. That's what "learning" means. So before training, you hide some data:

  • The training set — the examples the model is allowed to study.
  • The test set — locked in a drawer, used once at the very end to grade it. The final exam it never got to peek at.

Now the gap between the two scores becomes a confession:

  • Underfit — bad on training and test. Too dumb to see the pattern. Tracing a curvy river with a single straight ruler.
  • Overfit — great on training, bad on test. It memorized the noise. The student who learned last year's exam word-for-word and melts when the wording changes.
  • Just right — good on training, nearly as good on test. Small gap.

You crank a polynomial up to degree 9 so it threads through every single training point — training error basically zero. What does it do on the test set?

"Fits the training data perfectly" and "generalizes" become opposites once you go too far. To pass through every noisy point the curve has to thrash up and down between them — and that thrashing is pure invention. New points don't sit on the noise the curve memorized, so they land in the gaps where the curve predicts nonsense. Training error keeps dropping as degree rises; test error bottoms out in the middle and then climbs. That U-shape in the test curve is the signature of overfitting, and you're about to watch it live.

"Resample data" draws fresh noise — use it to prove the pattern is real, not a fluke of one dataset. Don't just wiggle the slider. Run the three missions.

What should I have seen?
Degree 1: a straight line missing the bend — high train AND test error (underfit). Mid degrees (often 3–4): both errors low and close together (the sweet spot). High degrees (8–9): training error crashes toward zero while test error shoots back up as the curve grows wild oscillations between the points (overfit). Resampling redraws the noise but the U-shape in the test curve survives — that's the proof it's a real phenomenon, not one unlucky dataset.
03 · why you can't just always go flexible

The dartboard

So if more flexibility lowers training error, why not always crank it up? Because flexibility cuts both ways, and the cleanest picture for this is a dartboard. Imagine training your model many times on slightly different samples of data. Each trained model is one dart thrown at the bullseye (the true answer).

An overfit model is like which dartboard pattern?

C — that scatter is variance. An overfit model overreacts to whatever noise was in its training sample, so tiny changes in the data swing it dramatically: the darts land everywhere. Pattern A is the opposite failure — an underfit model is consistently wrong in the same direction (a tight cluster, but off-target). That consistent offset is bias. Pattern B, tight and centered, is the dream you rarely fully get.

So the two words mean:

  • Bias — how far, on average, your darts land from the bullseye. High bias = consistently off the same way. That's an underfit, too-simple model: same mistake no matter the data.
  • Variance — how spread out the darts are from each other. High variance = scatter that depends on which sample you happened to draw. That's an overfit model.

Green = the true target. Blue dots = models trained on different samples. Bias = distance from center; variance = spread.

The tradeoff: make a model more flexible and bias drops (it can fit more shapes) but variance rises (it overreacts to noise). Make it simpler and the reverse. The error you actually feel splits into three pieces:

\[ \text{Expected test error} = \text{Bias}^2 + \text{Variance} + \text{irreducible noise} \]

Bias systematic wrongness — the same mistake every time. Variance sensitivity to which training sample you got. irreducible noise randomness in the world no model can ever predict (a fair coin flip). The sweet spot is the complexity that minimizes the first two — exactly the bottom of the U you found in the lab.

04 · don't trust one split

Cross-validation: the test set, but five times

One train-test split has a quiet flaw: your score depends on which rows happened to land in the test pile. Unlucky split, you trash a good model. Lucky split, you over-praise a bad one. On small data this luck-of-the-draw can swing the number a lot.

k-fold cross-validation refuses to rely on one split. Chop the data into \( k \) equal chunks (folds; a common choice is \( k = 5 \)). Train and test \( k \) times. Each round one fold is the test set and the other \( k-1 \) folds are training. Every row gets to be a test row exactly once.

5-fold CV: every fold is the test set once. Average the 5 scores for a stable estimate.

\[ \text{CV score} = \frac{1}{k} \sum_{i=1}^{k} s_i \]

k the number of folds. s_i the score (say accuracy) when fold \( i \) was held out for testing. The final score is just the average of the \( k \) rounds.

Bonus: the spread of those \( k \) scores (their standard deviation) is its own warning light. A huge spread means the model is unstable — high variance — even if the average looks fine.

05 · the four buckets

Classification metrics: where accuracy goes to die

For a yes/no classifier every prediction lands in one of four buckets — the confusion matrix. Using fraud as "positive":

  • True Positive (TP): predicted positive, was positive. (Caught real fraud.)
  • False Positive (FP): predicted positive, was negative. (Declined a legit purchase — false alarm.)
  • False Negative (FN): predicted negative, was positive. (Missed real fraud.)
  • True Negative (TN): predicted negative, was negative. (Correctly let a legit purchase through.)

Three metrics fall out of those four counts, and each answers a different question:

\[ \text{Accuracy} = \frac{TP + TN}{TP + TN + FP + FN} \]

"Of all predictions, what fraction were correct?" Simple — and a liar on imbalanced data (remember the fraud opener).

\[ \text{Precision} = \frac{TP}{TP + FP} \]

"When the model says positive, how often is it right?" High precision = few false alarms. Care about it when a false positive is expensive — your real card getting declined at a restaurant.

\[ \text{Recall} = \frac{TP}{TP + FN} \]

(a.k.a. sensitivity) "Of all the actual positives, how many did we catch?" High recall = few misses. Care about it when a false negative is dangerous — a cancer screen that misses a real tumor.

\[ F_1 = 2 \cdot \frac{\text{Precision} \cdot \text{Recall}}{\text{Precision} + \text{Recall}} \]

One number balancing the two. It's the harmonic mean, which punishes the case where one is high and the other is low — you can't fake it by maxing only one. The leading 2 makes it equal 1 when both are perfect.

The hidden knob: the threshold. Most classifiers don't output yes/no — they output a probability, like 0.83. You pick a threshold (default 0.5): positive if above it. Lower the threshold and you catch more positives (recall up) but raise false alarms (precision down). The ROC curve plots true-positive rate vs false-positive rate as you sweep the threshold; AUC (area under it) collapses the whole curve into one number from 0.5 (random) to 1.0 (perfect), measuring how well the model ranks positives above negatives regardless of any one threshold.

06 · do it by hand

Worked example — predict, then compute

A spam classifier is tested on 100 emails ("spam" = positive):

  • TP = 40 (spam correctly caught)
  • FP = 10 (good email wrongly marked spam)
  • FN = 20 (spam that slipped through)
  • TN = 30 (good email correctly delivered)

Before computing anything: precision is \(40/50 = 0.80\) and recall is \(40/60 \approx 0.667\). Where will F1 land?

The harmonic mean always sits at or below the plain average, and it leans toward the weaker of the two. That's the entire point: F1 won't let you brag about precision while recall quietly tanks. Below it works out to 0.727 — under the arithmetic mean of 0.733, tugged down by the 0.667 recall.

Now the arithmetic, every fraction shown. Try each yourself first, then check.

\[ \text{Accuracy} = \frac{40 + 30}{40+30+10+20} = \frac{70}{100} = 0.70 \]

70% of emails classified correctly.

\[ \text{Precision} = \frac{40}{40+10} = \frac{40}{50} = 0.80 \]

When this filter says "spam," it's right 80% of the time.

\[ \text{Recall} = \frac{40}{40+20} = \frac{40}{60} = 0.6\overline{6} \approx 0.667 \]

We caught ~67% of the spam; the other 33% slipped into inboxes.

\[ F_1 = 2 \cdot \frac{0.80 \times 0.667}{0.80 + 0.667} = 2 \cdot \frac{0.5336}{1.467} = 2 \cdot 0.3637 \approx 0.727 \]

Accuracy 0.70, precision 0.80, recall 0.667, F1 0.727. F1 sits between precision and recall, pulled toward the smaller — exactly the balance you predicted.

07 · when you predict a number

Regression metrics: MAE vs RMSE

Predicting a price or a temperature instead of a class? Two metrics dominate. Let \( y_i \) be the true value, \( \hat{y}_i \) ("y-hat") the prediction, \( n \) the number of examples.

\[ \text{MAE} = \frac{1}{n}\sum_{i=1}^{n} |y_i - \hat{y}_i| \]

MAE (Mean Absolute Error) average size of the mistakes; an over- and under-prediction of the same size count equally. Forgiving of outliers, easy to say out loud ("on average we're off by $4,200").

\[ \text{RMSE} = \sqrt{\frac{1}{n}\sum_{i=1}^{n} (y_i - \hat{y}_i)^2} \]

RMSE (Root Mean Squared Error) square the errors first, average, then square-root back to original units. The squaring makes big mistakes hurt disproportionately — the metric to watch when one large error is much worse than several small ones.

You met squared error in Module 3 — Linear Regression as the thing you minimized; here it comes back as the thing you report.

Model A's errors are [2, 2, 2, 2, 2]. Model B's are [0, 0, 0, 0, 10]. Both have MAE = 2.0. Which metric exposes Model B's one giant mistake?

MAE averages raw error sizes, so it treats five mistakes of 2 the same as one mistake of 10 — both give 2.0. RMSE squares first: \( \sqrt{(0+0+0+0+100)/5} = \sqrt{20} \approx 4.47 \) for B versus \( \sqrt{20/5} = 2.0 \) for A. The squaring is what makes the single outlier scream. Pick RMSE when a rare huge error is the thing you're scared of.

08 · the lever against overfitting

Regularization: tax the loud weights

Overfitting usually shows up as huge weights. To thread through every noisy point, a model cranks its coefficients to extremes — that's where the wild wiggles in the lab came from. Regularization adds a penalty to the loss for being too loud, forcing a trade between fitting the data and keeping weights small.

\[ \text{Loss}_{\text{ridge}} = \underbrace{\frac{1}{n}\sum_{i=1}^{n}(y_i - \hat{y}_i)^2}_{\text{fit the data}} + \underbrace{\lambda \sum_{j=1}^{p} w_j^2}_{\text{stay small}} \]

L2 / Ridge. w_j the \( j \)-th weight, p the number of weights, λ ("lambda") the regularization strength you choose. \( \lambda = 0 \) is plain regression; as \( \lambda \) grows, the penalty dominates and weights are squeezed toward zero — simpler model, lower variance, higher bias. Ridge shrinks smoothly but rarely hits exactly zero.

\[ \text{Loss}_{\text{lasso}} = \frac{1}{n}\sum_{i=1}^{n}(y_i - \hat{y}_i)^2 + \lambda \sum_{j=1}^{p} |w_j| \]

L1 / Lasso. Same idea, but it penalizes the absolute values. The kink at zero lets it drive some weights exactly to zero — deleting those features outright. That makes Lasso a built-in feature selector, handy when you suspect many features are useless.

And you pick \( \lambda \) the proper way — with cross-validation — not by eyeballing it. Bias-variance, CV, and regularization aren't separate topics; they're one loop: CV measures the U, \( \lambda \) slides you along it.

09 · find the bug

This CV code looks careful. It cheats.

A diligent beginner scaled their features (good!) and cross-validated (good!). Their CV score comes back suspiciously high, and the model later flops in production. The code runs without errors. Find the bug before opening the answer.

import numpy as np from sklearn.preprocessing import StandardScaler from sklearn.linear_model import LogisticRegression from sklearn.model_selection import cross_val_score X, y = load_my_data() scaler = StandardScaler() X = scaler.fit_transform(X) # <-- scale EVERYTHING up front, before CV clf = LogisticRegression(max_iter=1000) scores = cross_val_score(clf, X, y, cv=5, scoring="f1") print(scores.mean()) # looks great... too great
Show the bug & the fix

This is data leakage. scaler.fit_transform(X) computes the mean and standard deviation from the entire dataset — including the rows that will later sit in each CV test fold. So when cross_val_score holds out a fold to "test" on, that fold already shaped the scaling. Information about the test data leaked into training, and the score comes back optimistically high. In production, where genuinely unseen data can't leak backward in time, performance drops.

Fix: put the scaler in a Pipeline so it is refit on each fold's training portion only:

from sklearn.pipeline import make_pipeline pipe = make_pipeline(StandardScaler(), LogisticRegression(max_iter=1000)) scores = cross_val_score(pipe, X, y, cv=5, scoring="f1") # honest now print(scores.mean())

Rule of thumb: any step that learns from the data (scaling, imputation, feature selection) must live inside the CV loop, never before it. More on this in Module 7 — Feature Engineering and Preprocessing.

10 · build it from scratch

The metrics are just four counts and arithmetic

No magic. Predict the printed line in your head before you run this — you already did the math above, so you know the four numbers.

def classification_metrics(tp, fp, fn, tn): accuracy = (tp + tn) / (tp + fp + fn + tn) precision = tp / (tp + fp) if (tp + fp) > 0 else 0.0 recall = tp / (tp + fn) if (tp + fn) > 0 else 0.0 if precision + recall > 0: f1 = 2 * precision * recall / (precision + recall) else: f1 = 0.0 return accuracy, precision, recall, f1 acc, prec, rec, f1 = classification_metrics(tp=40, fp=10, fn=20, tn=30) print(f"accuracy={acc:.3f} precision={prec:.3f} recall={rec:.3f} f1={f1:.3f}") # K-fold split indices by hand (no libraries) def kfold_indices(n_samples, k): indices = list(range(n_samples)) fold_size = n_samples // k folds = [] start = 0 for i in range(k): # last fold absorbs any remainder end = start + fold_size if i < k - 1 else n_samples test_idx = indices[start:end] train_idx = indices[:start] + indices[end:] folds.append((train_idx, test_idx)) start = end return folds for fold_num, (train_idx, test_idx) in enumerate(kfold_indices(10, k=5), 1): print(f"Fold {fold_num}: test={test_idx} train={train_idx}")

It prints accuracy=0.700 precision=0.800 recall=0.667 f1=0.727 — matching your hand calc exactly. The k-fold loop prints five rows; fold 1 tests on indices [0, 1] and trains on [2..9], fold 2 tests on [2, 3], and so on, so every index appears in a test set exactly once. (In practice you'd shuffle the indices first so the folds aren't biased by row order.)

The library way

scikit-learn does all of it in a few lines. Before running: cross_val_score returns one number per fold. How many numbers will the first print show? (Five — cv=5.) And as alpha grows, which loses more weights to zero, Ridge or Lasso?

import numpy as np from sklearn.datasets import make_classification, make_regression from sklearn.linear_model import LogisticRegression, Ridge, Lasso from sklearn.model_selection import cross_val_score, train_test_split from sklearn.metrics import confusion_matrix, classification_report # --- classification: cross-validation + confusion matrix --- X, y = make_classification(n_samples=400, n_features=8, weights=[0.7, 0.3], random_state=0) clf = LogisticRegression(max_iter=1000) scores = cross_val_score(clf, X, y, cv=5, scoring="f1") print(f"5-fold F1: {scores}") print(f"mean={scores.mean():.3f} std={scores.std():.3f}") X_tr, X_te, y_tr, y_te = train_test_split(X, y, test_size=0.25, random_state=0) clf.fit(X_tr, y_tr) pred = clf.predict(X_te) print("Confusion matrix [[TN FP] [FN TP]]:") print(confusion_matrix(y_te, pred)) print(classification_report(y_te, pred, digits=3)) # --- regression: Ridge vs Lasso as lambda (alpha) grows --- Xr, yr = make_regression(n_samples=200, n_features=20, n_informative=5, noise=10.0, random_state=0) for alpha in [0.1, 1.0, 10.0]: ridge = Ridge(alpha=alpha).fit(Xr, yr) lasso = Lasso(alpha=alpha).fit(Xr, yr) n_zero = int(np.sum(np.abs(lasso.coef_) < 1e-8)) print(f"alpha={alpha:>4}: ridge||w||={np.linalg.norm(ridge.coef_):.1f} " f"lasso zeros={n_zero}/20")

scikit-learn calls the regularization strength alpha (our \( \lambda \)). The CV block prints five F1 scores plus their mean and standard deviation — far more trustworthy than any single split. As alpha grows, Ridge's total weight magnitude shrinks, and Lasso zeroes out more of the 20 features (it should keep roughly the 5 truly informative ones) — L1's automatic feature selection in action.

13 · extra reps

A few more, with solutions hidden

  1. A model produces TP = 90, FP = 30, FN = 10, TN = 870. Compute accuracy, precision, recall, F1. Is accuracy a fair summary here?
    Show solution
    Accuracy = (90 + 870)/1000 = 0.960. Precision = 90/120 = 0.75. Recall = 90/100 = 0.90. F1 = 2·(0.75·0.90)/(0.75+0.90) ≈ 0.818. Accuracy looks great (0.96) but only 100 of 1000 are positive — the easy negatives inflate it. Precision/recall/F1 are the honest picture.
  2. 12 rows, 4-fold CV, no shuffling. List the test indices per fold, and say how many times each row is used for training.
    Show solution
    12/4 = 3 rows per fold. Fold 1 test = [0,1,2], Fold 2 = [3,4,5], Fold 3 = [6,7,8], Fold 4 = [9,10,11]. Each row is a test row exactly once, so it's a training row in the other k−1 = 3 rounds.
  3. For a hospital screening test, prioritize precision or recall? What happens to each as you lower the threshold?
    Show solution
    Prioritize recall: a false negative tells a sick patient they're healthy — potentially fatal — so catch as many true cases as possible, accepting more false alarms (follow-up tests rule those out). Lowering the threshold predicts "positive" more readily: recall goes up, precision goes down.
  4. Train Ridge with λ = 0, then push λ to a very large value. What happens to (a) the weights, (b) bias, (c) variance? At λ → ∞, what does the model predict?
    Show solution
    (a) Weights shrink toward zero as the penalty \( \lambda \sum w_j^2 \) outweighs the fit term. (b) Bias increases — simpler model, less able to follow the data. (c) Variance decreases — tiny weights react less to noise. As λ → ∞ all weights → 0, so the model ignores the features and just predicts the constant mean of the training targets (max bias, near-zero variance).
