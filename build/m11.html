Module 11

Training Deep Networks That Actually Work

Learning objectives

By the end of this module you'll be able to:

  • Use the core PyTorch pieces — tensors, autograd, and nn.Module — and explain what each one buys you over plain NumPy.
  • Write the canonical training loop from memory (forward → loss → backward → step → zero-grad) and say exactly what each line does.
  • Choose between optimizers (plain SGD, SGD with momentum, and Adam) and explain in words how each one decides its next step.
  • Apply the three pillars of "training that actually works" — regularization (dropout, weight decay), normalization (batch/layer norm), and good initialization — and say what problem each one solves.
  • Diagnose vanishing/exploding gradients, learning-rate trouble, and the train-vs-eval-mode bug, and use learning-rate schedules to train faster and better.
  • Handle the practical engineering: move a model and its data to the same device (CPU/GPU), save and load weights, and debug dimension-mismatch errors calmly.

Why should I care?

In Module 10 — Neural Network Fundamentals you built a network by hand and hand-derived backpropagation to see that there's no magic inside. That was the right way to learn the idea — and it is also nobody's idea of a good time when the network has a million parameters. Real practitioners do not hand-derive gradients. They describe the network, hand it some data, and a framework computes every gradient automatically and updates every weight. That framework, for most of the field, is PyTorch. Here's the part that surprises beginners: getting a network to learn at all is usually not about a clever architecture. It's about a dozen unglamorous details — the right optimizer, a sane learning rate, normalization so the signal doesn't fade, dropout so you don't memorize the training set, remembering to call optimizer.zero_grad(), and not accidentally leaving your model in training mode during evaluation. A team can have a brilliant model design that produces garbage purely because one of these knobs is wrong. This module is the field manual for those knobs. Master it and "my loss is nan and I don't know why" turns from a mystery into a five-minute fix.

Intuition first

Think back to Module 4 — Gradient Descent. To take one step downhill you need two things: the slope of the cost with respect to every parameter (the gradient), and a rule for turning that slope into a step. For a line with two knobs you wrote the gradient formula by hand. For a deep network with millions of knobs, writing those formulas by hand is hopeless. The single biggest thing PyTorch gives you is a robot that computes the entire gradient for you, exactly, no matter how complicated the network — a feature called autograd (short for "automatic differentiation").

The receipt that remembers itself. Imagine every arithmetic operation you do is written on a long receipt, in order: "multiplied a by b, then added c, then squared the result." When you finally ask "how does the last number depend on the very first input?", you don't re-derive anything — you just read the receipt backwards, applying the chain rule one line at a time, and the gradient falls out. PyTorch keeps exactly this receipt automatically as your code runs (it calls it the computation graph), and .backward() is the act of reading it backwards. You write the forward calculation once, in ordinary-looking Python, and you get the gradient for free. That is the whole trick that makes deep learning practical.

The second big idea is organization. A real network is a stack of layers, each with its own weights. PyTorch gives you nn.Module, a container that holds layers, knows about all their parameters, and runs them in order when you call it. You describe what the network computes in a forward method; PyTorch handles the bookkeeping of which numbers are learnable and how to update them.

With those two pieces, training collapses into a short, fixed ritual you will repeat for the rest of your ML life. Five steps, always in this order:

The fixed ritual: predict, score the prediction, read the receipt backwards to get gradients, take a downhill step, then wipe the gradients clean before the next batch. Skip the last box and your gradients pile up — the single most common beginner bug.

PyTorch essentials

A tensor is PyTorch's version of a NumPy array: a grid of numbers (a scalar, vector, matrix, or higher). It behaves almost exactly like a NumPy array — same indexing, same broadcasting, similar functions (torch.zeros, x.mean()) — with two superpowers added. First, a tensor can live on a GPU, where thousands of multiplications happen in parallel. Second, a tensor can track its own gradient: set requires_grad=True and PyTorch starts writing the receipt for every operation it takes part in.

A module (nn.Module) is a reusable bundle of computation plus its learnable parameters. nn.Linear(3, 1) is a module that holds a \(3\)-input, \(1\)-output linear layer — internally a weight matrix and a bias, both already marked requires_grad=True for you. You stack modules to build a network, and the parent module automatically discovers all the child parameters (so you never hand-collect them).

An optimizer is the object that owns the update rule. You hand it the list of parameters once (model.parameters()); thereafter optimizer.step() applies one update to every parameter using the gradients autograd just computed, and optimizer.zero_grad() resets those gradients to zero for the next round.

Making it precise

Autograd is just the chain rule from Module 1 — Just-Enough Math Intuition, applied mechanically. Suppose your network computes a loss \( L \) as a chain of operations from a parameter \( w \). The chain rule says: to get \( \frac{\partial L}{\partial w} \) (read "the partial derivative of the loss with respect to \( w \)" — how much the loss changes when you nudge \( w \)), you multiply together the local derivative at each step of the chain. If \( L \) depends on an intermediate value \( u \), which depends on \( w \), then

\[ \frac{\partial L}{\partial w} = \frac{\partial L}{\partial u} \cdot \frac{\partial u}{\partial w} \]

where each factor is a "local" derivative that's easy to compute on its own. For a deep chain you just keep multiplying local derivatives, one per operation, from the loss back to the parameter. That is exactly what .backward() does: it starts at the loss with a seed of \( 1 \), walks the receipt backwards, multiplies the local derivative at each node, and deposits the accumulated product into each tensor's .grad attribute. You never write the chain by hand; you just call one method.

Once gradients are in .grad, the optimizer applies an update. Plain stochastic gradient descent uses the same rule you learned in Module 4, written per-parameter \( \theta \) (theta, a stand-in for any weight or bias):

\[ \theta := \theta - \alpha \, \frac{\partial L}{\partial \theta} \]

where \( \alpha \) (alpha) is the learning rate, a small positive number you choose, and \( := \) means "becomes." That is all optimizer.step() does for plain SGD. The fancier optimizers below change only the right-hand side — how the raw gradient gets turned into a step.

Optimizers: SGD, momentum, and Adam

Plain SGD steps directly along the current gradient. The trouble: if the loss surface is a long narrow valley (steep across, gently sloped along), the raw gradient mostly points across the valley, so SGD zig-zags off the walls and crawls along the floor.

SGD with momentum fixes this by keeping a running average of recent gradients — a velocity — and stepping with that instead. Let \( v \) be the velocity and \( \beta \) (beta, typically \( 0.9 \)) the fraction of the old velocity we keep:

\[ v := \beta\, v + \frac{\partial L}{\partial \theta}, \qquad \theta := \theta - \alpha\, v \]

In words: each step is mostly "keep going the way I was going," nudged by the new gradient. The across-valley wobbles cancel out (they flip sign every step) while the along-valley push accumulates — like a heavy ball that ignores small bumps and builds speed downhill.

Adam ("adaptive moment estimation") goes one step further: it gives each parameter its own learning rate, automatically scaled by how big that parameter's gradients have recently been. A parameter with consistently tiny gradients gets a relatively larger step; one with huge swings gets a smaller, steadier step. The practical upshot for you: Adam usually just works with a default learning rate of \( 0.001 \), which is why it's the default choice for almost everything. The trade-off is that plain SGD with momentum, carefully tuned, sometimes generalizes a touch better — but as a beginner, reach for Adam first.

Regularization for deep nets: dropout and weight decay

Deep networks have so many parameters they can memorize the training set — fitting the noise, not the pattern — which is the overfitting you met in Module 6 — Model Evaluation. Two cheap, standard cures:

Dropout: during training, randomly "switch off" a fraction \( p \) of neurons (set their outputs to zero) on every forward pass. The network can't lean on any single neuron because it might vanish next batch, so it's forced to spread the work around and learn redundant, robust features. Crucially, dropout is only active during training; at evaluation time every neuron is used. This is exactly why train mode and eval mode differ — more on that in the pitfalls.

Weight decay: gently shrink every weight toward zero on each step by adding a penalty proportional to the size of the weights. This is the deep-learning name for the L2 regularization from Module 6: it discourages any one weight from growing huge, which keeps the function smooth. In PyTorch you turn it on with one argument: weight_decay=1e-4 in the optimizer.

Normalization: keeping the signal alive

As data flows through many layers, the scale of the numbers can drift — growing huge or shrinking to nothing — which makes training slow and brittle. Normalization layers re-center and re-scale the activations partway through the network so each layer receives inputs in a sane, predictable range (roughly mean \( 0 \), spread \( 1 \)), then let the network rescale them back if it wants via two learnable knobs. Batch normalization computes those statistics across the examples in the current mini-batch (great for vision, Module 12). Layer normalization computes them across the features of each single example (the standard in Transformers, Module 14, because it doesn't depend on batch size). You don't need the formulas to use them — just know that dropping a normalization layer between linear layers usually makes a deep net train faster and tolerate a higher learning rate.

Initialization and vanishing/exploding gradients

Before training starts, every weight needs a value. Set them all to zero and every neuron in a layer computes the same thing and receives the same gradient forever — they never differentiate (the "symmetry" problem). So we start with small random weights. But the scale of that randomness matters enormously. If weights are a bit too large, the signal (and its gradient) multiplies up layer after layer until it explodes to nan; a bit too small and it shrinks toward zero until the early layers get essentially no gradient and stop learning. These are the famous exploding and vanishing gradient problems. The fix is principled initialization (e.g. Kaiming/He init for ReLU networks, Xavier/Glorot for tanh), which sizes the random weights so the signal's spread stays roughly constant across layers. PyTorch's built-in layers already use sensible defaults, so this mostly matters when you build custom layers — but knowing the failure mode lets you recognize it instantly.

Learning-rate schedules

A single fixed learning rate is a compromise: large enough to make fast progress early, but then too large to settle gently into the minimum. A schedule lets the rate change over time — typically starting moderate and decaying toward zero. Common choices: step decay (multiply by \( 0.1 \) every few epochs), cosine annealing (smoothly ease the rate down following a cosine curve), and a brief warmup (start tiny and ramp up over the first few hundred steps, which stabilizes the volatile early phase — standard for Transformers). The intuition: take bold strides while you're far from the valley floor, then tiptoe as you arrive.

Worked example — by hand

Let's prove to ourselves that autograd computes a real, correct gradient by checking it against calculus we can do on paper. Take the simplest possible function with one input:

\[ y = (x - 2)^2 \]

We want the slope \( \frac{dy}{dx} \) at the point \( x = 5 \). By the chain rule (bring the power down, then multiply by the derivative of the inside, which is \( 1 \)):

\[ \frac{dy}{dx} = 2\,(x - 2) \cdot 1 = 2(x - 2) \]

Plug in \( x = 5 \):

\[ \left.\frac{dy}{dx}\right|_{x=5} = 2\,(5 - 2) = 2 \times 3 = 6 \]

So the by-hand answer is exactly \( 6 \). Let's read what this number means: at \( x = 5 \), nudging \( x \) up by a tiny amount makes \( y \) rise about \( 6 \) times as fast — the function is climbing steeply there (which makes sense, since the bottom of this parabola is at \( x = 2 \) and we're well to the right of it). Now we ask PyTorch the same question. We create \( x = 5 \) as a tensor that tracks gradients, compute \( y \), call y.backward(), and read x.grad:

import torch x = torch.tensor(5.0, requires_grad=True) # x = 5, and "watch its gradient" y = (x - 2) ** 2 # build the receipt: subtract 2, then square y.backward() # read the receipt backwards (chain rule) print(x.grad) # the slope dy/dx at x = 5

This prints tensor(6.) — identical to our hand calculation. Behind the scenes PyTorch recorded two operations (a subtraction and a squaring), and backward() multiplied their local derivatives: the squaring contributes \( 2(x-2) = 6 \) and the subtraction contributes \( 1 \), giving \( 6 \times 1 = 6 \). One tensor, one method call, and the chain rule you'd grind through by hand for a deep network is done for you. That equivalence — autograd is the chain rule, just automated — is the entire foundation of everything below.

In code: from scratch

Before we lean on the framework, let's see autograd doing the chain rule on a slightly longer computation, and confirm by hand. We'll compute \( z = (3w + 1)^2 \) at \( w = 2 \). By hand: \( \frac{dz}{dw} = 2(3w+1)\cdot 3 = 6(3w+1) \); at \( w = 2 \) that's \( 6 \times 7 = 42 \).

import torch w = torch.tensor(2.0, requires_grad=True) u = 3 * w + 1 # intermediate value u = 7 z = u ** 2 # z = 49 z.backward() print("z =", z.item()) # 49.0 print("dz/dw =", w.grad.item()) # 42.0 -> matches 6*(3*2+1) = 42 # A subtle, important point: gradients ACCUMULATE. # If we backprop again WITHOUT zeroing, grads add up. w2 = torch.tensor(2.0, requires_grad=True) for step in range(3): out = (3 * w2 + 1) ** 2 out.backward() print(f"after backward #{step+1}, w2.grad = {w2.grad.item()}") # prints 42, then 84, then 126 -- each call ADDS 42, it does not replace.

The first block prints z = 49.0 and dz/dw = 42.0, matching the hand calculation exactly. The second block reveals the reason the training loop must call zero_grad(): each backward() adds to .grad rather than overwriting it, so the printed gradients are \( 42, 84, 126 \). PyTorch accumulates on purpose (it's occasionally useful), but it means that in a normal loop you must wipe the gradients each iteration — otherwise step two uses the sum of step one's and step two's gradients, and your training quietly corrupts itself.

Now let's build a complete from-scratch training loop that uses autograd for the gradients but does the optimizer step by hand, so nothing is hidden. We'll fit \( y \approx 2x + 1 \), updating the weight and bias manually inside a torch.no_grad() block (which tells PyTorch "don't track these update operations — they're not part of the model's computation").

import torch torch.manual_seed(0) x = torch.linspace(0, 5, 50).reshape(-1, 1) # 50 inputs, shape (50, 1) y = 2.0 * x + 1.0 + 0.3 * torch.randn(x.shape) # true line + noise w = torch.zeros(1, 1, requires_grad=True) # weight, tracked b = torch.zeros(1, requires_grad=True) # bias, tracked lr = 0.05 for epoch in range(1000): yhat = x @ w + b # forward: matrix-multiply, shape (50, 1) loss = ((yhat - y) ** 2).mean() # mean squared error loss.backward() # autograd fills w.grad and b.grad with torch.no_grad(): # update WITHOUT recording on the receipt w -= lr * w.grad b -= lr * b.grad w.grad.zero_() # wipe gradients for the next epoch b.grad.zero_() if epoch % 200 == 0: print(f"epoch {epoch:4d} loss {loss.item():.4f} " f"w {w.item():.3f} b {b.item():.3f}") print(f"\nlearned w = {w.item():.3f}, b = {b.item():.3f} (true 2 and 1)")

The loss falls steadily and the parameters climb toward the truth, ending near \( w \approx 2 \), \( b \approx 1 \) (the floor the loss settles at is just the noise we added, which no model can fit). This is the canonical loop in its most transparent form: forward (compute predictions and loss), backward (autograd computes every gradient), step (move each parameter downhill), zero-grad (clear for next time). The only thing the library version below changes is that an optimizer object does the step-and-zero for us — the four phases are identical.

In code: the library way

Now the real thing: a complete, runnable PyTorch training loop on a small classification dataset, with everything a real project has — a Dataset and DataLoader for mini-batching, an nn.Module model with dropout, the Adam optimizer with weight decay, the canonical loop, train/eval mode switching, device handling, and a saved checkpoint. We use a small synthetic dataset so it runs in seconds anywhere (no download), but the structure is identical to what you'd write for MNIST — swap the dataset and you're training on real images.

import torch from torch import nn from torch.utils.data import TensorDataset, DataLoader torch.manual_seed(0) device = "cuda" if torch.cuda.is_available() else "cpu" # use GPU if present # --- 1. Data: two interleaving blobs -> a 2-feature, 2-class problem --- n = 1000 X = torch.randn(n, 2) # label 1 if inside a tilted region, else 0 (a nonlinear boundary) y = ((X[:, 0] * 1.5 - X[:, 1] + 0.4 * X[:, 0] ** 2) > 0).long() split = 800 train_ds = TensorDataset(X[:split], y[:split]) test_ds = TensorDataset(X[split:], y[split:]) train_dl = DataLoader(train_ds, batch_size=32, shuffle=True) # mini-batches test_dl = DataLoader(test_ds, batch_size=32) # --- 2. Model: a small MLP with a hidden layer, ReLU, and dropout --- class Net(nn.Module): def __init__(self): super().__init__() self.net = nn.Sequential( nn.Linear(2, 32), # 2 inputs -> 32 hidden units nn.ReLU(), nn.Dropout(0.2), # active only in train mode nn.Linear(32, 2), # 32 -> 2 class scores (logits) ) def forward(self, x): return self.net(x) model = Net().to(device) # model on the chosen device loss_fn = nn.CrossEntropyLoss() # correct loss for classification optimizer = torch.optim.Adam(model.parameters(), lr=1e-3, weight_decay=1e-4) # Adam + weight decay # --- 3. The canonical training loop --- for epoch in range(20): model.train() # turn dropout ON for xb, yb in train_dl: xb, yb = xb.to(device), yb.to(device) # data on the SAME device as model logits = model(xb) # forward loss = loss_fn(logits, yb) # loss loss.backward() # backward (autograd) optimizer.step() # update every parameter optimizer.zero_grad() # wipe grads for next batch # --- evaluate each epoch --- model.eval() # turn dropout OFF correct = total = 0 with torch.no_grad(): # no receipt -> faster, less memory for xb, yb in test_dl: xb, yb = xb.to(device), yb.to(device) preds = model(xb).argmax(dim=1) # pick the higher-scoring class correct += (preds == yb).sum().item() total += yb.size(0) if epoch % 5 == 0 or epoch == 19: print(f"epoch {epoch:2d} test accuracy {correct/total:.3f}") # --- 4. Save and reload the trained weights --- torch.save(model.state_dict(), "net.pt") # save just the weights (best practice) fresh = Net().to(device) fresh.load_state_dict(torch.load("net.pt", map_location=device)) fresh.eval() print("reloaded model ready")

This trains in a second or two and prints a test accuracy climbing into the low-to-mid \( 0.9 \)s as the network learns the curved boundary. Walk through the four phases inside the loop and you'll see they are exactly the from-scratch loop, just delegated: model(xb) is forward, loss_fn(...) is the loss, loss.backward() is autograd, optimizer.step() is the update, optimizer.zero_grad() is the wipe. Five extra things separate this from a toy: (1) DataLoader serves shuffled mini-batches; (2) model.train()/model.eval() flip dropout on and off; (3) torch.no_grad() during evaluation saves time and memory because we don't need gradients there; (4) every tensor is moved to the same device as the model; (5) state_dict() saves the weights so you can ship the model. That is a production-shaped training loop. You'll reuse this skeleton in Module 12 — CNNs and Module 13 — Sequences and NLP with only the model and dataset swapped.

Common pitfalls & misconceptions

  • Forgetting optimizer.zero_grad(). Gradients accumulate in PyTorch, so if you never zero them, each step uses the sum of all previous gradients. Training behaves bizarrely — usually the loss refuses to settle or diverges. The fix is one line per batch; the from-scratch section above shows the accumulation happening (42 → 84 → 126) so you can recognize the symptom.
  • The wrong loss for the task. Use nn.CrossEntropyLoss for classification and nn.MSELoss for regression — not the other way around. And a sharp edge: nn.CrossEntropyLoss expects raw logits (it applies softmax internally), so do not add a softmax in your model's final layer, and the targets must be integer class indices, not one-hot vectors. Getting this wrong produces a model that trains but learns nothing sensible.
  • Learning rate too high or too low. Same lesson as Module 4, now with deep nets: a loss that jumps to nan means the rate is too high — divide it by 10. A loss that barely moves means it's too low (or you forgot to scale your inputs). Start with Adam at lr=1e-3 and adjust from there.
  • Leaving the model in the wrong mode. model.train() keeps dropout active and batch-norm updating its statistics; model.eval() turns dropout off and freezes those statistics. Evaluating while still in train mode gives noisy, randomly-worse numbers; training while in eval mode disables your regularization. Always set the mode explicitly before each phase — it is the most common "my test accuracy is mysteriously unstable" bug.
  • Model and data on different devices. If the model is on the GPU (.to("cuda")) but a batch is still on the CPU, PyTorch raises Expected all tensors to be on the same device. Move both the model (once, at creation) and every batch (inside the loop) to the same device variable.
  • Vanishing or exploding gradients in deep nets. A very deep network whose loss is stuck (gradients near zero, early layers frozen) or blowing up to nan often has an initialization/normalization problem, not a bug in your loop. Reach for normalization layers (batch/layer norm), a sane activation (ReLU rather than sigmoid in hidden layers), and, if needed, gradient clipping (torch.nn.utils.clip_grad_norm_) to cap the step size.
  • Reading a metric off a tensor with its gradient attached. Use loss.item() (a plain Python number) when logging, and accumulate metrics inside torch.no_grad(). Summing raw loss tensors across a whole epoch keeps the entire computation graph alive and silently exhausts your memory.

Exercises

  1. By hand, find \( \frac{dy}{dx} \) for \( y = (2x - 1)^2 \) at \( x = 3 \). Then write the three lines of PyTorch that confirm it, and state what x.grad will print.
    Show solution

    Chain rule: \( \frac{dy}{dx} = 2(2x-1)\cdot 2 = 4(2x-1) \). At \( x = 3 \): \( 4(2\cdot3 - 1) = 4 \times 5 = 20 \). The code: x = torch.tensor(3.0, requires_grad=True), then y = (2*x - 1)**2, then y.backward(). After that, x.grad prints tensor(20.), matching the hand answer.

  2. Put the five steps of the canonical training loop in the correct order from this scrambled list: optimizer.step(), loss = loss_fn(pred, y), optimizer.zero_grad(), pred = model(x), loss.backward(). Then explain what breaks if zero_grad() is left out.
    Show solution

    Correct order: (1) pred = model(x) — forward; (2) loss = loss_fn(pred, y) — loss; (3) loss.backward() — compute gradients; (4) optimizer.step() — update parameters; (5) optimizer.zero_grad() — clear gradients. (Placing zero_grad() at the very start of the next iteration is equally valid; the rule is just that it must come between successive backward() calls.) If it's left out, gradients from every batch accumulate, so each step moves by the sum of all gradients seen so far — the effective step size grows without bound and training destabilizes or diverges.

  3. You're building a 3-class classifier. A friend's model ends with nn.Linear(64, 3) followed by nn.Softmax(dim=1), and uses nn.CrossEntropyLoss. What's wrong, and how do you fix it?
    Show solution

    nn.CrossEntropyLoss already applies softmax (well, log-softmax) internally and expects raw logits. Adding an explicit nn.Softmax applies softmax twice, squashing the scores into a flat, near-uniform range that gives weak, poorly-scaled gradients — the model trains slowly and badly. Fix: delete the nn.Softmax layer; let the final nn.Linear(64, 3) output raw logits straight into the loss. If you need probabilities at inference time, apply softmax only then, outside the loss.

  4. A teammate reports: "I get RuntimeError: Expected all tensors to be on the same device, but found at least two devices, cuda:0 and cpu." Their model was created with model = Net().to("cuda"). Diagnose the bug and give the one-line fix inside the loop.
    Show solution

    The model lives on the GPU but the input batch is still on the CPU, so the first layer's weights (cuda) and the data (cpu) can't be multiplied. The fix is to move each batch to the same device inside the loop, e.g. xb, yb = xb.to(device), yb.to(device) right after you unpack the batch from the DataLoader. Both the model (once, at creation) and every batch (every iteration) must be on the same device.

  5. You train a deep network and the loss goes 0.7 → 0.69 → 0.69 → 0.69 … and never improves, and you confirm the learning rate isn't the issue. List three structural changes you'd try and explain the reasoning behind each.
    Show solution

    A loss flat from the start often means almost no useful gradient is reaching the early layers — a vanishing-gradient / initialization symptom. Three fixes: (1) Add normalization (e.g. nn.BatchNorm1d between linear layers) to keep activations and gradients at a healthy scale across depth. (2) Use ReLU instead of sigmoid/tanh in hidden layers — sigmoids saturate (their slope goes to zero for large inputs), choking the gradient; ReLU passes gradient through unattenuated for positive inputs. (3) Check/fix initialization — if you wrote custom layers, apply Kaiming (He) init so the signal's spread stays constant across layers rather than vanishing. A fourth option for the opposite failure (loss → nan) is gradient clipping. Each targets the same root cause: keeping a usable gradient flowing from the loss all the way back to the first layer.
