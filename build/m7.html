Module 7

Feature Engineering and Preprocessing

Learning objectives

By the end of this module you'll be able to:

  • Scale numbers so a model treats them fairly — using standardization (z-scores) and min-max normalization — and explain exactly why this speeds up gradient descent.
  • Turn raw columns into more useful ones with numerical transforms, polynomial features, and interaction features.
  • Convert text categories into numbers correctly with one-hot and ordinal encoding, and recognize when target encoding causes data leakage.
  • Handle missing values with imputation and deal sensibly with outliers.
  • Engineer real features on a messy dataset (the Titanic passenger list) by hand and with scikit-learn.
  • Build a leak-free scikit-learn Pipeline with a ColumnTransformer that fits on training data only.

Why should I care?

Imagine you're predicting whether a loan will be repaid. Your spreadsheet has an income column in dollars (values like 45000) and an age column in years (values like 27). To a learning algorithm that only sees raw numbers, income looks roughly 1,600 times "bigger" than age — so it will obsess over income and barely notice age, purely because of the units someone happened to type in. Worse, the table has blanks where applicants skipped a field, a "home ownership" column full of words like "rent" and "own" that no equation can multiply, and a few suspicious entries claiming an income of 99,999,999. None of the beautiful models from Modules 3–6 can touch this data as-is. Feature engineering is the unglamorous, high-leverage work of turning a raw, messy table into clean numbers that let a model actually learn. Practitioners routinely find that better features beat a fancier algorithm — and this is the module where you learn to make them.

Intuition first

A model never sees your data the way you do. It sees a grid of numbers and tries to find weights to multiply them by. "Feature engineering" is everything you do to that grid before the model touches it, so that the patterns you care about become easy to find. There are four jobs in this module, and they all serve that one goal.

Job 1: Put numbers on a common scale. Suppose two features are "age in years" (around 20–60) and "salary in dollars" (around 30,000–100,000). In Module 4 you learned that gradient descent walks downhill on a cost surface. When one feature is thousands of times larger than another, that surface becomes a long, skinny valley instead of a round bowl. Gradient descent then zig-zags down the narrow valley, taking tiny careful steps — slow and frustrating. Rescale both features to a comparable range and the valley becomes a nice round bowl: gradient descent walks straight to the bottom.

The recipe analogy. Picture a recipe written half in grams and half in truckloads. "Add 200 grams of flour and 0.0001 truckloads of salt." Technically correct, but your brain (and a model) struggles to weigh ingredients measured in wildly different units. Convert everything to grams first and suddenly the relative amounts are obvious. Scaling does exactly this: it rewrites every feature in comparable "units" so the model can judge their importance on the merits, not on the accident of measurement.

Left: with unscaled features the contours are stretched, so gradient descent (green) zig-zags. Right: after scaling the contours are circular and the path is a straight shot to the minimum.

Job 2: Create features the model can't invent on its own. A linear model can only add up weighted inputs; it cannot, by itself, multiply two columns together or square one. If the true pattern is "risk is high only when someone is both young and has high debt," you can hand the model that idea by building a new column equal to age × debt. That's an interaction feature. Squaring a column (a polynomial feature) lets a straight-line model bend into a curve. You're doing some of the model's thinking for it.

Job 3: Turn words into numbers. A column like color with values "red", "blue", "green" means nothing to arithmetic. If you naively map them to 1, 2, 3 you've secretly told the model that green (3) is "three times" red (1) and that blue (2) sits exactly between them — a false ordering. One-hot encoding avoids this by giving each category its own yes/no column.

Job 4: Patch holes and tame extremes. Real data has blanks (missing values) and freak values (outliers). You'll fill the blanks sensibly (imputation) and decide what to do about the freaks, because both can quietly wreck a model.

One last framing that will pay off later: feature engineering is the classical machine-learning ancestor of prompt engineering. In both, you can't change the model's internal machinery much, so you change its input to make the right answer easy to produce. Crafting an age × debt column for linear regression and crafting a careful instruction for a large language model are the same instinct — shape the input so the model succeeds. You'll meet the LLM side in Module 14 — Transformers and LLMs.

Making it precise

Standardization (z-scores)

Standardization rescales a feature so it has mean 0 and standard deviation 1. For a single feature with values \( x \), we compute

\[ z = \frac{x - \mu}{\sigma} \]

where \( x \) is one raw value, \( \mu \) (mu) is the mean of that feature (the average of all its values), and \( \sigma \) (sigma) is the standard deviation (a measure of how spread out the values are). In plain words: subtract the average to recenter the data around zero, then divide by the spread so the typical distance from zero becomes about 1. The result \( z \) is the "z-score" — how many standard deviations a value sits above (positive) or below (negative) the mean.

The standard deviation itself is \( \sigma = \sqrt{\frac{1}{n}\sum_{i=1}^{n}(x_i-\mu)^2} \), where \( n \) is the number of values, \( x_i \) is the \( i \)-th value, and the big \( \sum \) (sigma sum) just means "add up over all values." Inside: take each value's distance from the mean, square it (so negatives don't cancel positives), average those squared distances, then take the square root to get back to the original units. This is the same population standard deviation that scikit-learn's StandardScaler uses.

Min-max normalization

Min-max squeezes every value into the range from 0 to 1:

\[ x' = \frac{x - x_{\min}}{x_{\max} - x_{\min}} \]

where \( x_{\min} \) and \( x_{\max} \) are the smallest and largest values of the feature. The smallest value maps to 0, the largest to 1, and everything else lands proportionally in between. Use min-max when you need a bounded range (some neural-network inputs, image pixels); use standardization when the data has outliers or you don't need a hard 0–1 box, because one extreme value can crush all the others toward 0 under min-max but only shifts the mean and spread under standardization.

One-hot encoding

For a categorical feature with \( k \) distinct categories, one-hot encoding replaces it with \( k \) new columns (one per category). Each row gets a 1 in the column for its category and 0 everywhere else. The category "red" with categories {red, blue, green} becomes the row \( (1, 0, 0) \). No false ordering is introduced because no category is numerically larger than another.

The golden rule: fit on train only

Every transform above has parameters it must learn from data: standardization needs \( \mu \) and \( \sigma \); min-max needs \( x_{\min} \) and \( x_{\max} \); imputation needs the fill value; one-hot needs the list of categories. You must compute these from the training set only, then apply the exact same numbers to the test set. If you compute \( \mu \) over the whole dataset, information from the test set leaks into training — exactly the data leakage you met in Module 2 — The ML Mindset, and your reported accuracy becomes a lie. We call this "fit on train, transform on both."

Worked example — by hand

Part A — standardize the list \( [10, 20, 30] \).

Step 1, the mean: \( \mu = \frac{10 + 20 + 30}{3} = \frac{60}{3} = 20 \).

Step 2, the squared distances from the mean: \( (10-20)^2 = 100 \), \( (20-20)^2 = 0 \), \( (30-20)^2 = 100 \).

Step 3, the variance (average of those): \( \frac{100 + 0 + 100}{3} = \frac{200}{3} \approx 66.67 \).

Step 4, the standard deviation: \( \sigma = \sqrt{66.67} \approx 8.165 \).

Step 5, the z-scores \( z = \frac{x-20}{8.165} \):

  • \( z_{10} = \frac{10-20}{8.165} = \frac{-10}{8.165} \approx -1.22 \)
  • \( z_{20} = \frac{20-20}{8.165} = 0 \)
  • \( z_{30} = \frac{30-20}{8.165} = \frac{10}{8.165} \approx +1.22 \)

So \( [10, 20, 30] \) becomes \( [-1.22,\ 0,\ 1.22] \). Notice the shape is preserved (evenly spaced, symmetric around the middle) but the units are gone — these are now "standard deviations from the mean."

Part B — one-hot encode the colors \( [\text{red}, \text{blue}, \text{red}] \).

The distinct categories are {blue, red} (two of them). We make one column per category, in alphabetical order, and place a 1 where the row matches:

rowcoloris_blueis_red
1red01
2blue10
3red01

One word column became two numeric columns the model can multiply by weights. (If a third color "green" had appeared, you'd have an is_green column too — three columns for three categories.)

In code: from scratch

Let's implement standardization and one-hot encoding ourselves with NumPy and pandas, so nothing is hidden. We deliberately compute the mean and std on a "train" array and reuse them on a "test" array, to make the fit-on-train rule concrete.

import numpy as np import pandas as pd # --- Standardization from scratch --- train = np.array([10.0, 20.0, 30.0]) test = np.array([25.0, 40.0]) # new, unseen values mu = train.mean() # learned from TRAIN only sigma = train.std() # population std (divides by n) print("mu =", mu, " sigma =", round(sigma, 3)) train_z = (train - mu) / sigma # transform train test_z = (test - mu) / sigma # transform test with SAME mu, sigma print("train standardized:", np.round(train_z, 2)) print("test standardized:", np.round(test_z, 2)) # --- One-hot encoding from scratch with pandas --- colors = pd.Series(["red", "blue", "red"], name="color") onehot = pd.get_dummies(colors, prefix="color").astype(int) print(onehot)

This prints mu = 20.0 sigma = 8.165, then train standardized: [-1.22 0. 1.22] — exactly our hand calculation. The test values become [ 0.61 2.45]: 25 is a little above the training mean, and 40 is far above it (about 2.45 training-standard-deviations up). Critically, we never recomputed the mean on the test set. The one-hot block prints two columns, color_blue and color_red, matching the worked example.

In code: the library way

scikit-learn gives every transform a .fit() method (learn the parameters) and a .transform() method (apply them). fit_transform does both at once and belongs on the training data; on test data you call transform only. Below we build a realistic end-to-end preprocessor with a ColumnTransformer (different handling per column) wrapped in a Pipeline (chain steps so they fit together and never leak).

import pandas as pd from sklearn.compose import ColumnTransformer from sklearn.pipeline import Pipeline from sklearn.preprocessing import StandardScaler, OneHotEncoder from sklearn.impute import SimpleImputer from sklearn.model_selection import train_test_split from sklearn.linear_model import LogisticRegression # A tiny messy dataset: some numbers, some categories, some blanks. df = pd.DataFrame({ "age": [22, 38, None, 26, 35, 58, 40, None], "fare": [7.25, 71.3, 7.92, 53.1, 8.05, 51.9, 21.0, 13.0], "city": ["NYC", "LA", "NYC", None, "LA", "SF", "SF", "NYC"], "survived": [0, 1, 1, 1, 0, 0, 1, 0], }) X = df.drop(columns="survived") y = df["survived"] # Split FIRST, so nothing learns from the test rows (Module 2 leakage rule). X_train, X_test, y_train, y_test = train_test_split( X, y, test_size=0.25, random_state=0) numeric = ["age", "fare"] categorical = ["city"] # Numbers: fill blanks with the (training) median, then standardize. num_pipe = Pipeline([ ("impute", SimpleImputer(strategy="median")), ("scale", StandardScaler()), ]) # Categories: fill blanks with the most frequent value, then one-hot. # handle_unknown="ignore" makes unseen test categories all-zeros instead of crashing. cat_pipe = Pipeline([ ("impute", SimpleImputer(strategy="most_frequent")), ("onehot", OneHotEncoder(handle_unknown="ignore")), ]) pre = ColumnTransformer([ ("num", num_pipe, numeric), ("cat", cat_pipe, categorical), ]) model = Pipeline([("pre", pre), ("clf", LogisticRegression())]) model.fit(X_train, y_train) # fits imputers/scaler/encoder on TRAIN only print("test accuracy:", model.score(X_test, y_test))

Because everything lives inside one Pipeline, calling model.fit(X_train, y_train) learns the medians, the mean/std, and the category list from the training rows alone; model.score(X_test, y_test) then applies those frozen numbers to the test rows. There is no way to accidentally leak, which is the whole point of pipelines. (This toy set is tiny, so don't read into the exact accuracy — the structure is the lesson.)

Hands-on: feature engineering on messy data (Titanic)

The Titanic passenger list is the classic playground for this skill. Three engineered features reliably help predict survival, and each shows a different technique.

import pandas as pd # Pretend we've loaded the Titanic CSV into a DataFrame `df` with columns like: # Name, Sex, Age, SibSp (siblings/spouses), Parch (parents/children), Fare. df = pd.DataFrame({ "Name": ["Braund, Mr. Owen", "Cumings, Mrs. John", "Heikkinen, Miss. Laina", "Allen, Master. William"], "SibSp": [1, 1, 0, 0], "Parch": [0, 0, 0, 1], "Fare": [7.25, 71.28, 7.92, 21.07], }) # 1) Title extracted from Name: "Mr", "Mrs", "Miss", "Master" carry age/sex/class signal. df["Title"] = df["Name"].str.extract(r",\s*([^.]+)\.") # Group rare titles so we don't one-hot a hundred near-empty columns. common = ["Mr", "Mrs", "Miss", "Master"] df["Title"] = df["Title"].where(df["Title"].isin(common), other="Rare") # 2) Family size: traveling alone vs. in a group changes survival odds. df["FamilySize"] = df["SibSp"] + df["Parch"] + 1 # +1 counts the passenger df["IsAlone"] = (df["FamilySize"] == 1).astype(int) # 3) Fare bins: turn a skewed continuous fare into ordered buckets. df["FareBin"] = pd.qcut(df["Fare"], q=2, labels=["low", "high"]) print(df[["Title", "FamilySize", "IsAlone", "FareBin"]])

You've created four new columns from raw text and counts: Title (a category extracted with a regular expression, then trimmed so rare titles collapse into "Rare"), FamilySize and IsAlone (interaction-style features combining two raw columns), and FareBin (a continuous number bucketed into ordered groups with pd.qcut, which splits at quantiles so each bin holds roughly equal counts). None of these existed in the raw data; each encodes a human hunch about what matters, handed to the model as a ready-made column. That is feature engineering.

A note on encoding choices and leakage

Ordinal encoding maps categories to ordered integers and is correct only when a real order exists — e.g. shirt size {S, M, L} → {0, 1, 2}. Use it for genuinely ordered categories; never for unordered ones like city or color.

Target (mean) encoding replaces each category with the average target value for that category — e.g. replace "city = SF" with the survival rate of SF passengers. It can be powerful for high-cardinality columns (thousands of categories), but it is a leakage magnet: if you compute those averages using the row's own target, the model effectively peeks at the answer, and accuracy looks great until it collapses on new data. If you ever reach for it, compute the averages with cross-validation folds so a row is never encoded using its own label. As a beginner, prefer one-hot until you understand exactly why you'd switch.

Common pitfalls & misconceptions

  • Scaling before the split. If you call fit_transform on the whole dataset and then split, the test set's mean and spread leaked into the scaler. Always split first, fit the scaler on train, transform both. A Pipeline enforces this automatically.
  • One-hot dummy explosion. A column with thousands of categories (like zip code) becomes thousands of columns, most of them almost all zeros — slow, memory-hungry, and prone to overfitting. Group rare categories into "Other," or use target encoding with proper cross-validation.
  • Unseen categories at test time. If "SF" appears only in the test set, a naive encoder crashes or invents a column the model never trained on. Set handle_unknown="ignore" in OneHotEncoder so unknown values become all-zeros.
  • Scaling tree-based models unnecessarily. Decision trees and random forests (Module 8 — Classical Algorithms) split on thresholds like "age > 30," which are unaffected by rescaling. Scaling them wastes effort and can confuse you about what helped. Scale for linear models, logistic regression, neural nets, and distance-based methods; skip it for trees.
  • Ordinal-encoding unordered categories. Mapping {red, blue, green} → {0, 1, 2} secretly tells the model green is "bigger" than red. Use one-hot for unordered categories; reserve ordinal for true orders.
  • Imputing with the test set's statistics. Filling missing values with the median computed over all data leaks test information. Learn the median on train, apply it to both — again, what a Pipeline guarantees.

Exercises

  1. Standardize the list \( [2, 4, 6, 8] \) by hand. Give \( \mu \), \( \sigma \), and the four z-scores.
    Show solution

    Mean: \( \mu = (2+4+6+8)/4 = 20/4 = 5 \). Squared distances: \( (2-5)^2=9,\ (4-5)^2=1,\ (6-5)^2=1,\ (8-5)^2=9 \), summing to 20. Variance: \( 20/4 = 5 \). Std: \( \sigma=\sqrt{5}\approx 2.236 \). Z-scores: \( (2-5)/2.236\approx -1.34 \), \( (4-5)/2.236\approx -0.45 \), \( (6-5)/2.236\approx 0.45 \), \( (8-5)/2.236\approx 1.34 \). So \( [-1.34, -0.45, 0.45, 1.34] \).

  2. Apply min-max normalization to \( [10, 20, 30] \). Then say what min-max value the unseen test value 40 would get, using the train min and max.
    Show solution

    Train min = 10, max = 30, range = 20. \( (10-10)/20=0 \), \( (20-10)/20=0.5 \), \( (30-10)/20=1 \). So \( [0, 0.5, 1] \). The test value 40: \( (40-10)/20 = 30/20 = 1.5 \). Note it lands above 1 — min-max does not clip, and unseen values can fall outside [0, 1]. That's expected and fine; just don't assume the range is guaranteed for new data.

  3. One-hot encode the column \( [\text{cat}, \text{dog}, \text{cat}, \text{bird}] \). How many columns, and what does the "dog" row look like?
    Show solution

    Three distinct categories {bird, cat, dog} → three columns is_bird, is_cat, is_dog (alphabetical). The rows: cat → (0,1,0); dog → (0,0,1); cat → (0,1,0); bird → (1,0,0). The "dog" row is \( (0, 0, 1) \).

  4. You have features height_cm (range 150–200) and weight_kg (range 45–120), and you'll train logistic regression with gradient descent. Explain in two or three sentences what goes wrong without scaling and how standardization fixes it.
    Show solution

    Without scaling, weight spans about 75 units while height spans about 50, and more importantly both are large compared to a 0–1 target signal, so the cost surface is an elongated valley. Gradient descent zig-zags slowly across the narrow direction and needs a tiny learning rate to stay stable. Standardizing both to mean 0, std 1 makes their scales comparable, rounding the cost contours so gradient descent heads almost straight to the minimum and converges in far fewer steps.

  5. A teammate writes: scaler.fit_transform(X); X_train, X_test = split(X). Identify the bug, explain the consequence, and write the corrected order in words (or pseudocode is fine).
    Show solution

    The bug: the scaler is fit on the full dataset before splitting, so the test rows' mean and standard deviation leak into the scaling parameters applied to the training rows. The consequence is optimistic, unreliable test scores — the evaluation no longer simulates truly unseen data. Correct order: split first (X_train, X_test = split(X)), then scaler.fit(X_train) to learn \( \mu, \sigma \) from train only, then scaler.transform(X_train) and scaler.transform(X_test) with those same numbers. Best practice: put the scaler in a Pipeline so this can't be done wrong.
