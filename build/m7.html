Version A trains slowly, fixates on income because the raw number is huge, chokes on the word "rent" (you can't multiply a weight by "rent"), and silently breaks on the blank cells. Version B converges fast and actually uses age. Nothing about the data changed. Only the features did.

That gap — same rows, wildly different model — is where amateurs lose and pros win. There is a folk theorem in applied ML: better features beat a fancier algorithm, almost every time. Most beginners spend weeks tuning models. The pros spend that time on the columns. This module is how you join the second group.

By the end you will scale numbers so a model treats them fairly, build features a linear model can't invent on its own, turn words into numbers without lying to the model, and wire it all into a leak-free scikit-learn Pipeline. But first, a question almost everyone gets wrong.

02 · commit before you read

Does scaling even matter? Depends who's asking.

"Scale your features" gets repeated like gospel. But when it helps depends entirely on the model. Lock in a guess for each.

You're training gradient descent (linear / logistic regression, a neural net) on income (~45,000) and age (~27). You DON'T scale. What happens?

When one feature is thousands of times bigger than another, the bowl from Module 4 stretches into a long skinny ravine. Gradient descent zig-zags across the narrow direction instead of walking straight down — slow, fragile, learning-rate-sensitive. Scale both features to a comparable range and the ravine rounds back into a bowl. Straight shot to the bottom. Scaling is the single highest-leverage one-liner for gradient-descent models.

Now you train a decision tree on the same unscaled income and age. Does scaling help?

A tree (and a random forest, Module 8) asks yes/no threshold questions: is this value above some cut point? Rescaling moves the cut point by the exact same amount, so the split — and the whole tree — comes out identical. Scaling a tree is wasted effort that can also confuse you about what actually helped. Rule of thumb: scale for gradient-descent and distance-based models; skip it for trees. "Always scale everything" is a beginner reflex, not a rule.

The recipe analogy. A recipe written half in grams and half in truckloads — "200 grams of flour, 0.0001 truckloads of salt" — is technically correct but impossible to reason about. Convert everything to grams and the relative amounts jump out. Scaling rewrites every feature in comparable units so a gradient-descent model judges importance on the merits, not on the accident of what units someone typed in.

03 · now the notation

You felt it. Here's how to actually do it.

The most common scaler is standardization: rewrite each feature so it has mean 0 and standard deviation 1. The rewritten value is a "z-score" — how many standard deviations a value sits above (+) or below (−) the mean.

\[ z = \frac{x - \mu}{\sigma} \]

x one raw value. μ (mu) the mean of the feature — its average. σ (sigma) the standard deviation — how spread out the values are. z the result: distance from the mean, measured in standard deviations.

In words: subtract the average to recenter on zero, then divide by the spread so the typical distance from zero becomes about 1. Units vanish; shape stays.

\[ \sigma = \sqrt{\tfrac{1}{n}\sum_{i=1}^{n}(x_i-\mu)^2} \]

n how many values. xi the i-th value. ∑ "add up over all values." Inside: each value's distance from the mean, squared (so negatives don't cancel), averaged, then square-rooted back into the original units. This is the population std — exactly what scikit-learn's StandardScaler uses.

The other scaler you'll meet is min-max normalization, which squeezes every value into [0, 1]:

\[ x' = \frac{x - x_{\min}}{x_{\max} - x_{\min}} \]

xmin, xmax the smallest and largest values of the feature. Smallest maps to 0, largest to 1, everything else lands proportionally between.

Which one? Use min-max when you need a hard bounded range (image pixels, some neural-net inputs). Use standardization when there are outliers or you don't need a 0–1 box — one freak value crushes everything else toward 0 under min-max, but only nudges the mean and spread under standardization.

04 · you do it first

Standardize [10, 20, 30] — by hand, before the answer.

Grab paper. Compute the mean, then the standard deviation, then the three z-scores. Predict all three z-scores before you reveal. (Hint: the list is symmetric — what should the middle z-score be?)

DO the full standardization, step by step

Don't peek. Even a wrong number you committed to teaches more than a right one you read.

Reveal the worked steps

Step 1 — mean: \( \mu = \frac{10+20+30}{3} = \frac{60}{3} = 20 \).

Step 2 — squared distances: \( (10-20)^2=100 \), \( (20-20)^2=0 \), \( (30-20)^2=100 \).

Step 3 — variance (average them): \( \frac{100+0+100}{3} = \frac{200}{3} \approx 66.67 \).

Step 4 — std: \( \sigma = \sqrt{66.67} \approx 8.165 \).

Step 5 — z-scores \( z = \frac{x-20}{8.165} \):

  • \( z_{10} = \frac{-10}{8.165} \approx -1.22 \)
  • \( z_{20} = \frac{0}{8.165} = 0 \)
  • \( z_{30} = \frac{10}{8.165} \approx +1.22 \)

So \( [10, 20, 30] \rightarrow [-1.22,\ 0,\ 1.22] \). The shape is preserved — evenly spaced, symmetric — but the units are gone. These are now "standard deviations from the mean."

You standardized your training set: income got μ=20, σ=8.165. Now a test value of 40 arrives. What z-score do you give it?

You learn μ and σ from train only, then freeze them and apply the same numbers to test. The test value 40 becomes ≈ 2.45 — "far above the training average." Recomputing on test (A) or per-set (C) lets test information bleed into your pipeline. That's data leakage (Module 2 — The ML Mindset), and it turns your reported accuracy into a comforting lie. The mantra: fit on train, transform on both.

05 · words into numbers

The trap that quietly poisons half of beginner models.

A column color with values "red", "blue", "green" means nothing to arithmetic. The tempting fix is to map them to 1, 2, 3. Do not.

You map {red, blue, green} → {1, 2, 3} and feed it to a linear model. What did you just secretly tell the model?

The model multiplies that number by a weight, so 1/2/3 becomes a real magnitude. You invented an order (green > blue > red) and a spacing that don't exist. The fix is one-hot encoding: give each category its own yes/no column. No category is numerically bigger than another, so no false order leaks in.

For a categorical feature with \( k \) distinct categories, one-hot makes \( k \) new columns. Each row gets a 1 in its category's column and 0 everywhere else. Your turn — do it on paper first.

DO one-hot encode [red, blue, red]

How many columns? What does row 2 look like? Write it before revealing.

One word column became two numeric columns a model can multiply by weights. Add a "green" and you'd get an is_green column too — three columns for three categories.

Two cousins, used rarely and carefully. Ordinal encoding maps categories to ordered integers — correct only when a real order exists, like shirt size {S, M, L} → {0, 1, 2}. Never for city or color. Target (mean) encoding replaces each category with the average target for that category (e.g. "city = SF" → SF's repayment rate). Powerful for high-cardinality columns, but a leakage magnet: if a row is encoded using its own label, the model peeks at the answer — accuracy looks great until it collapses on new data. If you must, compute the averages with cross-validation folds so a row is never encoded with its own label. As a beginner: prefer one-hot until you know exactly why you'd switch.

06 · do the model's thinking for it

A linear model can only add. Hand it the rest.

A linear model adds up weighted inputs. That's all. It cannot, on its own, multiply two columns or square one. So if the real pattern needs that, you build the column yourself:

  • Interaction feature: if risk is high only when someone is both young and deep in debt, hand the model age × debt as a new column. It could never have formed that product itself.
  • Polynomial feature: squaring a column (age²) lets a straight-line model bend into a curve.
  • Bucketing & combining: turn a messy continuous column into ordered bins, or fold two raw counts into one meaningful count.

You're doing part of the model's thinking for it. Which is the whole game.

The same instinct, later. Feature engineering is the classical ancestor of prompt engineering. In both, you can't easily change the model's internals — so you reshape its input to make the right answer easy to produce. Crafting an age × debt column for linear regression and crafting a careful instruction for an LLM are the same move. You'll meet the LLM side in Module 14 — Transformers & LLMs. When you get there, remember: you already know how to do this.

08 · find the bug

This code runs clean, reports great accuracy, and is lying to you.

No error. No crash. The test score looks fantastic — and it's worthless. Find the bug before the reveal. (Hint: ask who computed the scaler's mean and std, and from which rows.)

from sklearn.preprocessing import StandardScaler from sklearn.model_selection import train_test_split from sklearn.linear_model import LogisticRegression scaler = StandardScaler() X_scaled = scaler.fit_transform(X) # something is wrong on THIS line X_train, X_test, y_train, y_test = train_test_split( X_scaled, y, test_size=0.25, random_state=0) model = LogisticRegression().fit(X_train, y_train) print("test accuracy:", model.score(X_test, y_test)) # looks amazing... and is fake
Show the bug & the fix

scaler.fit_transform(X) computes the mean and std over the whole dataset — including the test rows — before the split. The test set's statistics leak into the numbers applied to training. The model has, in a quiet way, already seen the test distribution, so its reported accuracy is optimistic and won't survive real unseen data. This is the exact "data leakage" from Module 2, hiding inside one innocent line.

The fix — split first, fit the scaler on train only, transform both:

X_train, X_test, y_train, y_test = train_test_split( X, y, test_size=0.25, random_state=0) scaler = StandardScaler() X_train_s = scaler.fit_transform(X_train) # learn mu, sigma from TRAIN X_test_s = scaler.transform(X_test) # apply the SAME numbers to test model = LogisticRegression().fit(X_train_s, y_train) print("test accuracy:", model.score(X_test_s, y_test))

Better still: put the scaler in a Pipeline so it is structurally impossible to leak. That's Beat 9.

09 · the leak-proof way

ColumnTransformer + Pipeline: leakage becomes impossible.

Every scikit-learn transform has .fit() (learn the parameters) and .transform() (apply them). fit_transform does both — and belongs on training data only; on test you call transform alone. A ColumnTransformer applies different handling per column; a Pipeline chains steps so they fit together and never leak. Wrap the model inside it and the golden rule enforces itself.

import pandas as pd from sklearn.compose import ColumnTransformer from sklearn.pipeline import Pipeline from sklearn.preprocessing import StandardScaler, OneHotEncoder from sklearn.impute import SimpleImputer from sklearn.model_selection import train_test_split from sklearn.linear_model import LogisticRegression # A tiny messy dataset: some numbers, some categories, some blanks. df = pd.DataFrame({ "age": [22, 38, None, 26, 35, 58, 40, None], "fare": [7.25, 71.3, 7.92, 53.1, 8.05, 51.9, 21.0, 13.0], "city": ["NYC", "LA", "NYC", None, "LA", "SF", "SF", "NYC"], "survived": [0, 1, 1, 1, 0, 0, 1, 0], }) X = df.drop(columns="survived") y = df["survived"] # Split FIRST, so nothing learns from the test rows (Module 2 leakage rule). X_train, X_test, y_train, y_test = train_test_split( X, y, test_size=0.25, random_state=0) numeric = ["age", "fare"] categorical = ["city"] # Numbers: fill blanks with the (training) median, then standardize. num_pipe = Pipeline([ ("impute", SimpleImputer(strategy="median")), ("scale", StandardScaler()), ]) # Categories: fill blanks with the most frequent value, then one-hot. # handle_unknown="ignore" makes unseen test categories all-zeros instead of crashing. cat_pipe = Pipeline([ ("impute", SimpleImputer(strategy="most_frequent")), ("onehot", OneHotEncoder(handle_unknown="ignore")), ]) pre = ColumnTransformer([ ("num", num_pipe, numeric), ("cat", cat_pipe, categorical), ]) model = Pipeline([("pre", pre), ("clf", LogisticRegression())]) model.fit(X_train, y_train) # fits imputers/scaler/encoder on TRAIN only print("test accuracy:", model.score(X_test, y_test))

Because everything lives inside one Pipeline, model.fit(X_train, y_train) learns the medians, the mean/std, and the category list from the training rows alone; model.score(X_test, y_test) applies those frozen numbers to the test rows. There is no way to accidentally leak — that is the entire point of pipelines. (This toy set is tiny; don't read into the exact accuracy. The structure is the lesson.)

Hands-on: engineer features on messy data (Titanic)

The Titanic passenger list is the classic playground. Three engineered features reliably help predict survival, each showing a different technique. Predict what Title will hold for "Cumings, Mrs. John" before running.

import pandas as pd # Pretend we've loaded the Titanic CSV into a DataFrame `df` with columns like: # Name, Sex, Age, SibSp (siblings/spouses), Parch (parents/children), Fare. df = pd.DataFrame({ "Name": ["Braund, Mr. Owen", "Cumings, Mrs. John", "Heikkinen, Miss. Laina", "Allen, Master. William"], "SibSp": [1, 1, 0, 0], "Parch": [0, 0, 0, 1], "Fare": [7.25, 71.28, 7.92, 21.07], }) # 1) Title extracted from Name: "Mr", "Mrs", "Miss", "Master" carry age/sex/class signal. df["Title"] = df["Name"].str.extract(r",\s*([^.]+)\.") # Group rare titles so we don't one-hot a hundred near-empty columns. common = ["Mr", "Mrs", "Miss", "Master"] df["Title"] = df["Title"].where(df["Title"].isin(common), other="Rare") # 2) Family size: traveling alone vs. in a group changes survival odds. df["FamilySize"] = df["SibSp"] + df["Parch"] + 1 # +1 counts the passenger df["IsAlone"] = (df["FamilySize"] == 1).astype(int) # 3) Fare bins: turn a skewed continuous fare into ordered buckets. df["FareBin"] = pd.qcut(df["Fare"], q=2, labels=["low", "high"]) print(df[["Title", "FamilySize", "IsAlone", "FareBin"]])

Four new columns from raw text and counts: Title (a category pulled out with a regular expression, then trimmed so rare titles collapse into "Rare"), FamilySize and IsAlone (interaction-style features combining two raw columns), and FareBin (a continuous number bucketed into ordered groups with pd.qcut, which splits at quantiles so each bin holds roughly equal counts). None existed in the raw data. Each encodes a human hunch about what matters, handed to the model as a ready-made column. That is feature engineering.

12 · the traps, named

The mistakes that look fine until they don't.

  • Scaling before the split. The bug you just debugged. fit_transform on the whole set leaks test statistics. Split first; a Pipeline enforces it.
  • Scaling a tree for no reason. Trees and forests (Module 8 — Classical Algorithms) split on thresholds — rescaling changes nothing and just muddies what actually helped.
  • One-hot dummy explosion. A zip-code column becomes thousands of near-empty columns — slow, memory-hungry, overfit-prone. Group rare levels into "Other," or use cross-validated target encoding.
  • Unseen categories at test time. A category that appears only in test crashes a naive encoder. Set handle_unknown="ignore" so it becomes all-zeros.
  • Ordinal-encoding unordered categories. {red, blue, green} → {0,1,2} smuggles in a fake order. One-hot for unordered; ordinal only for true orders.
  • Imputing with the test set's median. Same leak, different transform. Learn the fill value on train, apply to both — what a Pipeline guarantees.
Two more hand exercises (with solutions)

1. Standardize [2, 4, 6, 8] by hand. Give μ, σ, and the four z-scores.

μ = (2+4+6+8)/4 = 20/4 = 5. Squared distances: 9, 1, 1, 9 → sum 20. Variance: 20/4 = 5. σ = √5 ≈ 2.236. Z-scores: (2−5)/2.236 ≈ −1.34, (4−5)/2.236 ≈ −0.45, (6−5)/2.236 ≈ 0.45, (8−5)/2.236 ≈ 1.34. So [−1.34, −0.45, 0.45, 1.34].

2. Min-max [10, 20, 30], then map the unseen test value 40.

Train min = 10, max = 30, range = 20. (10−10)/20 = 0, (20−10)/20 = 0.5, (30−10)/20 = 1 → [0, 0.5, 1]. Test 40: (40−10)/20 = 30/20 = 1.5. It lands above 1 — min-max doesn't clip, and unseen values can fall outside [0, 1]. Expected and fine; just don't assume the range holds for new data.
