Project Track — Five Increasing-Difficulty Projects

What this track is for

The modules taught you the ideas. These five projects turn ideas into skill. Each one is a self-contained, portfolio-worthy build that uses the concepts from the modules right before it. By the end you will have shipped: a classifier with honest evaluation, a model bake-off on real tabular data, a deployed image recognizer, a pretrained-language-model app, and your own document question-answering bot. That is a genuine ML resume.

  • Practice the full workflow — load data, split it honestly, train, evaluate, iterate — not just one isolated step.
  • Build five artifacts you can show people: notebooks, a Gradio web app, and a RAG bot over your own files.
  • Develop judgment: when to reach for a random forest vs. a linear model, when accuracy lies, when a model is just memorizing.
  • Learn to pair-program with your Claude Code companion the way working engineers actually use AI assistants — to scaffold, debug, and review, not to hand you finished answers.

Why projects beat tutorials (escaping "tutorial hell")

"Tutorial hell" is the trap where you watch and copy along, everything makes sense in the moment, and then you sit in front of a blank file and freeze. It feels like learning because the code runs and the dopamine hits — but you were following turn-by-turn GPS, not building a map in your head. The cure is uncomfortable on purpose: start from a blank file and a goal, get stuck, and work your way out. The struggle is the learning. A tutorial hands you the answer; a project forces you to form the question.

Each project below gives you a goal, a real dataset, milestones, and measurable success criteria — but it deliberately does not give you a complete copy-paste solution. You write the bulk of it. When you get stuck (you will, that is the point), you debug it yourself first, then ask your companion to explain the error rather than to silently fix it. Aim to be able to rebuild each project from a blank file the second time. If you can do that, it is yours.

Pair-program these with your Claude Code companion

Your Claude Code companion lives in your terminal and can read your files, run your code, and reason about errors. Use it like a patient senior engineer sitting next to you — to unblock and review, not to do the project for you. Good habits:

  • Scaffold, then fill it in yourself: "Give me a commented skeleton (function names and TODOs only, no implementations) for a train/validate/test pipeline on a tabular dataset. I'll write the bodies."
  • Explain my error, don't fix it: "Here's my traceback and the relevant code. Explain what's causing it and give me a hint, but let me write the fix."
  • Review like a mentor: "Review this notebook for data leakage, an unfair train/test split, or metrics that could mislead me. List issues with severity; don't rewrite it."
  • Quiz me on my own project: "Ask me 5 questions about why I made the modeling choices in this file, and push back if my answers are hand-wavy."
  • Generate practice variations: "Suggest 3 small experiments I could run on this dataset to test whether my model is overfitting, and predict what each will show."

Rule of thumb: if you could not re-explain a line your companion wrote, you have not learned it yet — ask it to slow down and teach that line.

Project 1 — Honest Classifier (do after Module 6)

Your first end-to-end project. The whole point is not "get high accuracy" — it is to evaluate a model honestly so you can trust the number you report. You will reuse the splitting, cross-validation, and confusion-matrix ideas from Module 6 — Model Evaluation.

Goal. Train a classifier on a small tabular dataset, evaluate it with a proper train/validation/test split and cross-validation, and report a confusion matrix plus precision/recall — not just accuracy.

Suggested dataset. Start with the Iris dataset (3 flower species, 150 rows, 4 features) via sklearn.datasets.load_iris — it is tiny and clean, perfect for getting the pipeline right. Then repeat the exact same pipeline on the messier Kaggle Titanic dataset (predict who survived), which forces you to handle missing values and categorical columns. Doing both teaches you that the process stays the same even when the data gets harder.

Milestones.

  1. Load the data and look at it: shapes, class balance, a few rows. Never model data you have not eyeballed.
  2. Split into train / validation / test. A clean recipe: hold out the test set first (e.g. 20%) and do not touch it again until the very end.
  3. Train a simple baseline (logistic regression or a decision tree) on the training set.
  4. Use k-fold cross-validation on the training data to estimate performance without peeking at test, and to compare two model choices fairly.
  5. Pick your final model, train it on all the training data, then evaluate once on the held-out test set.
  6. Produce a confusion matrix and a classification report (precision, recall, F1 per class). Write 3 sentences interpreting it: which classes get confused, and would accuracy alone have hidden that?

How you know you succeeded. You can state your test accuracy and per-class precision/recall; your test set was never used for any decision before the final evaluation; your cross-validation scores are reported with a mean and spread (e.g. 0.96 ± 0.03); and you can explain, pointing at the confusion matrix, exactly which mistakes your model makes.

Stretch goals. Run the same pipeline on Titanic with a preprocessing step (impute missing ages, one-hot encode categoricals — preview of Module 7). Plot a decision boundary using just two Iris features. Compare a decision tree against logistic regression using cross-validation and decide which you would deploy.

Starter code — a correct skeleton for the Iris half. Fill in the interpretation and then port it to Titanic yourself.

import numpy as np from sklearn.datasets import load_iris from sklearn.model_selection import train_test_split, cross_val_score from sklearn.linear_model import LogisticRegression from sklearn.metrics import confusion_matrix, classification_report # 1) Load and inspect data = load_iris() X, y = data.data, data.target print("Shape:", X.shape, "| Classes:", np.bincount(y), "| Names:", list(data.target_names)) # 2) Hold out a test set we will NOT touch until the end. # stratify=y keeps the class proportions the same in both splits. X_train, X_test, y_train, y_test = train_test_split( X, y, test_size=0.20, random_state=42, stratify=y) # 3) Cross-validate on the TRAINING data to estimate performance honestly. model = LogisticRegression(max_iter=1000) cv_scores = cross_val_score(model, X_train, y_train, cv=5) print(f"5-fold CV accuracy: {cv_scores.mean():.3f} +/- {cv_scores.std():.3f}") # 4) Train on all training data, then evaluate ONCE on the untouched test set. model.fit(X_train, y_train) y_pred = model.predict(X_test) print("\nConfusion matrix (rows = true, cols = predicted):") print(confusion_matrix(y_test, y_pred)) print("\n", classification_report(y_test, y_pred, target_names=data.target_names)) # YOUR TURN: write 3 sentences interpreting the confusion matrix. # Which two species get confused? Would plain accuracy have hidden that?

Expected output: 5-fold CV accuracy around 0.95–0.97, and a near-perfect test confusion matrix with the only mistakes between versicolor and virginica (the two species that genuinely overlap). That last fact is the lesson: the model's errors are not random, and the confusion matrix reveals where in a way a single accuracy number never could.

Project 2 — Model Bake-Off on Real Tabular Data (do after Module 8)

Now you compare model families and discover the workhorses of real-world tabular ML: random forests and gradient boosting. Uses Module 7 — Feature Engineering and Module 8 — Classical Algorithms.

Goal. On both a regression task and a classification task, build a fair head-to-head comparison of (a) a linear/logistic model, (b) a random forest, and (c) gradient boosting — with sensible feature engineering — and decide which wins and why.

Suggested dataset. For regression, use the California Housing dataset (predict median house value from 8 features) via sklearn.datasets.fetch_california_housing. For classification, reuse Titanic from Project 1, or pick any Kaggle tabular set you like. California Housing is ideal because the features (median income, house age, location) reward thoughtful engineering.

Milestones.

  1. Explore: distributions, correlations, missing values, obvious outliers. Note which features look predictive.
  2. Build a preprocessing pipeline (scaling for the linear model, imputation, encoding) using sklearn.pipeline.Pipeline so preprocessing is fit on training data only — this prevents data leakage.
  3. Engineer at least two new features (e.g. rooms-per-household, or interaction terms) and check whether they help.
  4. Train and cross-validate all three model families with the same splits and the same metric (RMSE for regression, F1 or ROC-AUC for classification).
  5. Tune the strongest model's key hyperparameters with a small grid or randomized search.
  6. Report a results table, inspect feature importances, and write a short conclusion: which model, which features mattered, and the cost/benefit (accuracy vs. interpretability vs. speed).

How you know you succeeded. You have one clean table comparing all three models on the same cross-validated metric; preprocessing is inside a pipeline so nothing leaks from validation into training; your tuned best model beats the linear baseline on the test set; and you can name the top 3 features driving predictions and explain why they make sense.

Stretch goals. Add XGBoost or LightGBM and see if they edge out scikit-learn's gradient boosting. Plot predicted vs. actual values and inspect where the model is most wrong. Use permutation importance instead of built-in importances and discuss the difference. Add a learning curve to diagnose whether more data would help (callback to Module 6).

Project 3 — Image Classifier, Deployed (do after Module 12)

Your first deep-learning system, end to end: a convolutional network you train, then a far stronger one you get for free via transfer learning, wrapped in a web app anyone can use. Uses Module 10–12 (neural nets, training, CNNs) and the deployment ideas from Module 15.

Goal. Train a CNN to classify images, beat your from-scratch CNN using a pretrained model (transfer learning), and deploy the result as an interactive Gradio web app where someone uploads an image and sees the prediction.

Suggested dataset. Start with MNIST handwritten digits (28×28 grayscale, 10 classes) via torchvision.datasets.MNIST — small, fast, trains in minutes even on CPU. Then level up to the Oxford-IIIT Pets dataset (37 cat/dog breeds, real color photos), where from-scratch training struggles and transfer learning shines — making the value of transfer learning unmistakable.

Milestones.

  1. Load the data with DataLoaders; visualize a batch so you trust your labels and normalization.
  2. Build and train a small CNN from scratch (a couple of conv layers + pooling + a linear head). Track train and validation accuracy per epoch.
  3. Swap in a pretrained model (e.g. ResNet-18 from torchvision.models), freeze the backbone, and train only a new classification head on your classes.
  4. Compare the two: accuracy, training time, and how many epochs each needed.
  5. Save the better model's weights and write a single predict(image) function.
  6. Wrap predict in a Gradio interface and launch it; upload your own images and watch it work.

How you know you succeeded. Your from-scratch CNN exceeds ~98% on MNIST test; on Pets, transfer learning clearly beats from-scratch with far less training; you can explain in plain words why a model pretrained on millions of images transfers to your 37 breeds; and the Gradio app correctly classifies an image you upload yourself.

Stretch goals. Add data augmentation (random flips/crops) and measure the gain. Unfreeze the top layers and fine-tune at a low learning rate. Show the model's confidence (softmax probabilities) in the app. Deploy the app publicly with share=True or to Hugging Face Spaces and send a friend the link.

Starter code — a correct from-scratch CNN on MNIST plus a Gradio app stub. Run it, then add the transfer-learning model yourself.

import torch import torch.nn as nn import torch.nn.functional as F from torch.utils.data import DataLoader from torchvision import datasets, transforms device = "cuda" if torch.cuda.is_available() else "cpu" # 1) Data: normalize to MNIST's known mean/std. tfm = transforms.Compose([transforms.ToTensor(), transforms.Normalize((0.1307,), (0.3081,))]) train_ds = datasets.MNIST(".", train=True, download=True, transform=tfm) test_ds = datasets.MNIST(".", train=False, download=True, transform=tfm) train_dl = DataLoader(train_ds, batch_size=128, shuffle=True) test_dl = DataLoader(test_ds, batch_size=256) # 2) A small CNN from scratch. class SmallCNN(nn.Module): def __init__(self): super().__init__() self.conv1 = nn.Conv2d(1, 16, 3, padding=1) # 1 -> 16 channels self.conv2 = nn.Conv2d(16, 32, 3, padding=1) # 16 -> 32 channels self.fc1 = nn.Linear(32 * 7 * 7, 64) self.fc2 = nn.Linear(64, 10) # 10 digit classes def forward(self, x): x = F.max_pool2d(F.relu(self.conv1(x)), 2) # 28 -> 14 x = F.max_pool2d(F.relu(self.conv2(x)), 2) # 14 -> 7 x = x.flatten(1) x = F.relu(self.fc1(x)) return self.fc2(x) # raw logits model = SmallCNN().to(device) opt = torch.optim.Adam(model.parameters(), lr=1e-3) loss_fn = nn.CrossEntropyLoss() # 3) Train for a few epochs. for epoch in range(3): model.train() for xb, yb in train_dl: xb, yb = xb.to(device), yb.to(device) opt.zero_grad() loss = loss_fn(model(xb), yb) loss.backward() opt.step() # Evaluate on the test set. model.eval() correct = total = 0 with torch.no_grad(): for xb, yb in test_dl: xb, yb = xb.to(device), yb.to(device) preds = model(xb).argmax(1) correct += (preds == yb).sum().item() total += yb.size(0) print(f"epoch {epoch}: test accuracy = {correct / total:.4f}") torch.save(model.state_dict(), "mnist_cnn.pt") # 4) Deploy with Gradio. (pip install gradio) import gradio as gr import numpy as np from PIL import Image def predict(img): # In current Gradio, a Sketchpad returns a dict; the drawing is under # "composite" (an RGBA HxWx4 array). Older versions returned a plain array, # so handle both. We then convert to a single 28x28 grayscale digit. if img is None: return {} if isinstance(img, dict): img = img.get("composite") if img is None: return {} arr = np.asarray(img) # MNIST is white digits on a black background. The sketchpad strokes are # opaque, so the alpha channel (where you drew) already gives that polarity. if arr.ndim == 3 and arr.shape[2] == 4: gray = arr[:, :, 3] # alpha channel = where you drew else: gray = np.asarray(Image.fromarray(arr.astype(np.uint8)).convert("L")) pil = Image.fromarray(gray.astype(np.uint8)).resize((28, 28)) x = torch.tensor(np.array(pil), dtype=torch.float32).reshape(1, 1, 28, 28) x = (x / 255.0 - 0.1307) / 0.3081 # same normalization as training model.eval() with torch.no_grad(): probs = F.softmax(model(x.to(device)), dim=1)[0] return {str(i): float(probs[i]) for i in range(10)} demo = gr.Interface(fn=predict, inputs=gr.Sketchpad(image_mode="RGBA"), outputs=gr.Label(num_top_classes=3), title="Draw a digit") demo.launch() # add share=True for a public link # YOUR TURN: add a ResNet-18 (torchvision.models.resnet18) transfer-learning # version on Oxford-IIIT Pets and compare accuracy + training time.

After 3 epochs the from-scratch CNN should hit roughly 98–99% test accuracy on MNIST, and the Gradio window lets you draw a digit and watch the top-3 probabilities update live. The lesson lands when you redo this on Pets: a from-scratch CNN flounders, while a frozen pretrained backbone with a fresh head reaches strong accuracy in a fraction of the time.

Project 4 — Text Classification with a Pretrained Language Model (do after Module 14)

You will use a pretrained Transformer the way industry does: download it, tokenize text, run it, and (optionally) fine-tune it on your task. Uses Module 13–14 (sequences, attention, Transformers) and the Hugging Face LLM Course.

Goal. Classify text (movie-review sentiment, or news-topic labels) using a pretrained Hugging Face model — first zero-shot/off-the-shelf via a pipeline, then by fine-tuning on labeled data — and measure how much fine-tuning buys you.

Suggested dataset. The IMDb movie-review dataset (50,000 reviews labeled positive/negative) via the Hugging Face datasets library: load_dataset("imdb"). It is the classic sentiment benchmark and downloads in one line.

Milestones.

  1. Load IMDb with the datasets library and inspect a few reviews and their labels.
  2. Run a baseline with the off-the-shelf pipeline("sentiment-analysis") on a sample and record its accuracy — no training at all.
  3. Understand tokenization: load the model's tokenizer, tokenize one sentence, and look at the input IDs and attention mask so you know what the model actually receives.
  4. (Fine-tune) Tokenize the dataset, load a small model like distilbert-base-uncased with a classification head, and train a few epochs with the Trainer API on a subset.
  5. Evaluate the fine-tuned model on a held-out test split and compare to the off-the-shelf baseline.
  6. Write a short report: did fine-tuning help, by how much, and on what kinds of reviews does the model still fail?

How you know you succeeded. You can explain what a tokenizer does and read the token IDs for a sentence; your off-the-shelf baseline and fine-tuned scores are both reported on the same test split; the fine-tuned model meets or beats the baseline (typically 90%+ on IMDb); and you can show two examples the model gets wrong and reason about why (sarcasm, mixed sentiment, length truncation).

Stretch goals. Try a multi-class topic dataset (e.g. AG News) and adapt the head. Compare two base models (DistilBERT vs. BERT) on accuracy and speed. Push your fine-tuned model to the Hugging Face Hub. Wrap it in a Gradio app like Project 3 so anyone can paste text and get a label.

Project 5 — Retrieval-Augmented Question Answering Over Your Own Docs (do after Module 14)

The capstone, and the architecture behind most real LLM products today. You will build a system that answers questions about your own documents by retrieving relevant chunks with embeddings and feeding them to a language model — Retrieval-Augmented Generation (RAG). Uses everything from Module 14 plus the embedding intuition from Module 9 / 13.

Goal. Build a small RAG question-answering app: index your own documents into a vector store, retrieve the most relevant passages for a question, and have a language model answer using only those passages — deployed via Gradio.

Suggested data. Use your own documents — a few PDFs, lecture notes, your notes from this course, a product manual, anything text. Real personal data makes the project unforgettable. Embeddings via sentence-transformers, vector search via FAISS (faiss-cpu).

Milestones.

  1. Load your documents and split them into overlapping chunks (a few hundred words each) so each chunk is a self-contained passage.
  2. Embed every chunk into a vector with a sentence-transformer and store the vectors in a FAISS index.
  3. For a user question, embed the question the same way and retrieve the top-k most similar chunks (nearest neighbors).
  4. Build a prompt that pastes the retrieved chunks as context and asks an LLM to answer using only that context (and to say "I don't know" if the answer isn't there).
  5. Wrap the whole flow in a Gradio chat interface.
  6. Test with questions whose answers are and are not in your docs; verify it refuses to hallucinate when it shouldn't know.

How you know you succeeded. Asking a question whose answer is in your docs returns a correct answer plus the source passage it used; asking something outside your docs makes it say it doesn't know rather than inventing an answer; and you can explain each stage — chunk, embed, index, retrieve, prompt — in one sentence apiece.

Stretch goals. Show the retrieved source chunks alongside each answer (citations). Compare chunk sizes and top-k values and observe the quality tradeoff. Swap the embedding model and measure retrieval quality. Add conversational memory so follow-up questions work. Deploy to Hugging Face Spaces.

Starter code — the retrieval core (chunk → embed → index → retrieve). Add your document loader and the LLM answer step on top.

# pip install sentence-transformers faiss-cpu import numpy as np import faiss from sentence_transformers import SentenceTransformer # 0) YOUR TURN: replace this with text loaded from your own PDFs / notes. documents = [ "The mitochondrion is the powerhouse of the cell, producing ATP.", "Photosynthesis converts sunlight into chemical energy in plants.", "Newton's second law states that force equals mass times acceleration.", "The capital of France is Paris, located on the river Seine.", "Gradient descent updates parameters in the direction of steepest descent.", ] # 1) Chunk: here each document is already one chunk. For real docs, split long # text into ~200-word overlapping windows so each chunk stands on its own. chunks = documents # 2) Embed every chunk into a vector. embedder = SentenceTransformer("all-MiniLM-L6-v2") embeddings = embedder.encode(chunks, normalize_embeddings=True) embeddings = np.asarray(embeddings, dtype="float32") # 3) Build a FAISS index. With normalized vectors, inner product = cosine sim. index = faiss.IndexFlatIP(embeddings.shape[1]) index.add(embeddings) # 4) Retrieve the top-k most relevant chunks for a question. def retrieve(question, k=2): q = embedder.encode([question], normalize_embeddings=True).astype("float32") scores, idx = index.search(q, k) return [(chunks[i], float(scores[0][j])) for j, i in enumerate(idx[0])] question = "How do cells make energy?" for text, score in retrieve(question): print(f"{score:.3f} {text}") # YOUR TURN: # - Build a prompt that pastes these retrieved chunks as context and asks an # LLM (Hugging Face pipeline, or an API) to answer USING ONLY that context, # and to reply "I don't know" if the answer is not present. # - Wrap retrieve(...) + the LLM call in a gradio.ChatInterface.

For the question "How do cells make energy?" the retriever should surface the mitochondrion and photosynthesis chunks with the highest similarity scores — even though the question shares no exact words with them. That is the magic of embeddings: they match on meaning, not keywords. Once you wire a language model on top to phrase the answer from those chunks, you have a working RAG system over your own knowledge.
