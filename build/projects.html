00 · the only test that matters

Close every tab. Open a blank file. Now build a spam filter.

Did your stomach drop? Good. That feeling — blank file, no tutorial to copy, no "next" button — is the exact gap between watching ML and doing ML. Everything you've studied so far has been training wheels. This is the part where you ride.

Here's the brutal truth about tutorials: they feel like learning because the code runs and the dopamine hits. But you were following turn-by-turn GPS, not building a map in your head. Take the GPS away and you're lost in a parking lot you "drove through" ten times. That trap has a name — tutorial hell — and the only escape is to start from a goal and a blank file, get stuck, and claw your way out. The struggle is the learning. So this track isn't five lessons. It's five quests, ramping in difficulty, each one a self-contained build you could put in front of a hiring manager. Beat all five and you've shipped: an honestly-evaluated classifier, a model bake-off on real data, a deployed image recognizer, a pretrained-language-model app, and a question-answering bot over your own documents. You watch a 6-hour "build an ML project" video, coding along the whole time, every cell green. The next day you open a blank file to do it solo. Most honest prediction?

Recognition is not recall. Following along lights up the "yeah, that makes sense" circuit and leaves the "produce it from nothing" circuit untouched — and only the second one shows up at a blank file or a job. 03 · Quest 1 of 5 · after Module 6

Quest 1 — The Honest Classifier

A model that reports 96% accuracy and a model you can trust are two different things. This quest is about the second one. Anyone can call .fit(); your job is to evaluate so honestly that you'd stake your name on the number.

The goal. Train a classifier on small tabular data and evaluate it the right way: a proper train / validation / test split and cross-validation, reported with a confusion matrix and per-class precision/recall — not a lonely accuracy number. Reuses the splitting, CV, and confusion-matrix ideas from Module 6 — Model Evaluation.

The data. Start with Iris (3 flower species, 150 rows, 4 features) via sklearn.datasets.load_iris — tiny and clean, perfect for getting the pipeline right. Then run the exact same pipeline on the messier Kaggle Titanic dataset (predict who survived), which forces you to handle missing values and categorical columns. Doing both teaches the real lesson: the process stays the same even when the data gets ugly.

LAB Quest 1 milestones

Check each off as you complete it in your own notebook. No skipping milestone 2 — the entire quest hinges on it.

  • Load and eyeball the data shapes, class balance, a few rows. Never model data you haven't looked at.
  • Split into train / validation / test hold out the test set first (~20%) and do NOT touch it again until the very end.
  • Train a simple baseline logistic regression or a decision tree, on the training set only.
  • k-fold cross-validate on the training data estimate performance without peeking at test, and compare two model choices fairly.
  • Final evaluation, exactly once retrain on all training data, then score on the untouched test set a single time.
  • Confusion matrix + classification report then write 3 sentences: which classes get confused, and would accuracy alone have hidden it?

How you know you won. You can state your test accuracy and per-class precision/recall; your test set was never used for any decision before the final score; your CV result is reported as mean ± spread (e.g. 0.96 ± 0.03); and you can point at the confusion matrix and name exactly which mistakes your model makes.

Stretch. Port the pipeline to Titanic with preprocessing (impute missing ages, one-hot encode categoricals — a preview of Module 7). Plot a decision boundary on two Iris features. Pit a decision tree against logistic regression with CV and decide which you'd deploy.

Starter code — a correct skeleton for the Iris half. Predict the printed output before you run it, then port it to Titanic yourself.

import numpy as np from sklearn.datasets import load_iris from sklearn.model_selection import train_test_split, cross_val_score from sklearn.linear_model import LogisticRegression from sklearn.metrics import confusion_matrix, classification_report # 1) Load and inspect data = load_iris() X, y = data.data, data.target print("Shape:", X.shape, "| Classes:", np.bincount(y), "| Names:", list(data.target_names)) # 2) Hold out a test set we will NOT touch until the end. # stratify=y keeps the class proportions the same in both splits. X_train, X_test, y_train, y_test = train_test_split( X, y, test_size=0.20, random_state=42, stratify=y) # 3) Cross-validate on the TRAINING data to estimate performance honestly. model = LogisticRegression(max_iter=1000) cv_scores = cross_val_score(model, X_train, y_train, cv=5) print(f"5-fold CV accuracy: {cv_scores.mean():.3f} +/- {cv_scores.std():.3f}") # 4) Train on all training data, then evaluate ONCE on the untouched test set. model.fit(X_train, y_train) y_pred = model.predict(X_test) print("\nConfusion matrix (rows = true, cols = predicted):") print(confusion_matrix(y_test, y_pred)) print("\n", classification_report(y_test, y_pred, target_names=data.target_names)) # YOUR TURN: write 3 sentences interpreting the confusion matrix. # Which two species get confused? Would plain accuracy have hidden that?

Expected: 5-fold CV accuracy around 0.95–0.97, and a near-perfect test confusion matrix whose only mistakes are between versicolor and virginica — the two species that genuinely overlap. That's the whole lesson in one observation: your model's errors aren't random, and the confusion matrix shows you where in a way a single accuracy number never can.

04 · Quest 2 of 5 · after Module 8

Quest 2 — The Model Bake-Off

Beginners ask "which algorithm is best?" Engineers ask "best on this data, by this metric, measured fairly?" This quest makes you run the bake-off and discover the quiet workhorses of real-world tabular ML: random forests and gradient boosting.

The goal. On a regression task, build a fair head-to-head between (a) a linear model, (b) a random forest, and (c) gradient boosting — with sensible feature engineering — and decide which wins and why. Uses Module 7 — Feature Engineering and Module 8 — Classical Algorithms.

The data. California Housing (predict median house value from 8 features) via sklearn.datasets.fetch_california_housing. It's ideal because the features — median income, house age, location — reward thoughtful engineering. For a classification version, reuse Titanic from Quest 1 or grab any Kaggle tabular set you like.

LAB Quest 2 milestones

The trap in this quest is leakage. Milestone 2 is what keeps the comparison honest — do it before you touch a model.

  • Explore the data distributions, correlations, missing values, obvious outliers. Note which features look predictive.
  • Build a preprocessing Pipeline scaling, imputation, encoding via sklearn.pipeline.Pipeline, fit on training data only — this is what prevents leakage.
  • Engineer at least two new features e.g. rooms-per-household or an interaction term, then check whether they actually help.
  • Cross-validate all three families fairly same splits, same metric — RMSE for regression, F1 or ROC-AUC for classification.
  • Tune the strongest model a small grid or randomized search on its key hyperparameters.
  • Results table + feature importances + conclusion which model, which features mattered, and the accuracy vs. interpretability vs. speed tradeoff.

How you know you won. One clean table compares all three models on the same cross-validated metric; preprocessing lives inside a pipeline so nothing leaks from validation into training; your tuned best model beats the linear baseline on the test set; and you can name the top 3 features driving predictions and explain why they make sense.

Stretch. Add XGBoost or LightGBM and see if they edge out scikit-learn's gradient boosting. Plot predicted vs. actual and find where the model is most wrong. Use permutation importance instead of built-in importances and discuss the difference. Add a learning curve to diagnose whether more data would help (callback to Module 6).

DEBUG One line leaks the test set into training. Find it before you peek.

A teammate's "fair" bake-off reports a suspiciously great score. It runs clean, no errors — and it's cheating. Read it. The bug isn't a typo; it's a workflow error.

from sklearn.datasets import fetch_california_housing from sklearn.model_selection import train_test_split, cross_val_score from sklearn.preprocessing import StandardScaler from sklearn.linear_model import LinearRegression X, y = fetch_california_housing(return_X_y=True) # "Standardize everything up front so all models see clean features." scaler = StandardScaler() X = scaler.fit_transform(X) # <-- something is wrong on this line X_train, X_test, y_train, y_test = train_test_split( X, y, test_size=0.2, random_state=0) model = LinearRegression() print(cross_val_score(model, X_train, y_train, cv=5, scoring="neg_root_mean_squared_error"))
Show the bug & why

scaler.fit_transform(X) is fit on the entire dataset — including the test rows — before the split. The scaler's mean and standard deviation are computed using information from data you're pretending you haven't seen yet. That's data leakage: your test score is now optimistically biased, and the whole bake-off is rigged. Fix: put scaling inside a Pipeline (or fit the scaler on X_train only and merely transform the test set). With a pipeline, cross-validation re-fits the scaler on each fold's training portion automatically — which is exactly milestone 2. Leakage is the single most common way a beginner's "amazing" model quietly lies.

05 · Quest 3 of 5 · after Module 12

Quest 3 — The Deployed Image Classifier

First you'll train a convolutional net by hand. Then you'll get a far stronger one almost for free — and the gap between those two is the most important practical idea in deep learning. Then you'll ship it as a web app a stranger can use.

The goal. Train a CNN to classify images, then beat your from-scratch CNN using a pretrained model (transfer learning), and deploy the result as an interactive Gradio web app where someone uploads or draws an image and sees the prediction. Uses Modules 10–12 (neural nets, training, CNNs) plus deployment from Module 15.

The data. Start with MNIST handwritten digits (28×28 grayscale, 10 classes) via torchvision.datasets.MNIST — small, fast, trains in minutes even on CPU. Then level up to Oxford-IIIT Pets (37 cat/dog breeds, real color photos), where from-scratch training struggles and transfer learning shines — making its value impossible to miss.

You have ~3,700 pet photos (37 breeds, 100 each). You train a CNN from scratch vs. fine-tune a ResNet-18 pretrained on millions of generic internet images. What happens?

3,700 photos is nowhere near enough to learn good visual features from zero — the from-scratch net overfits or flounders. The pretrained backbone already learned the universal low-level vocabulary of vision (edges, fur texture, ear shapes) from millions of images; you just teach a fresh final layer to map those features to your 37 breeds. That's transfer learning, and on small datasets it's not a small win — it's the difference between "barely works" and "ships." You'll feel this firsthand when MNIST-from-scratch is fine but Pets-from-scratch isn't.

LAB Quest 3 milestones

Milestone 1 (visualize a batch) is non-optional — half of all deep-learning bugs are silently wrong labels or normalization you'd have caught by looking.

  • Load data with DataLoaders and visualize a batch so you trust your labels and normalization before training a thing.
  • Train a small CNN from scratch a couple of conv layers + pooling + a linear head. Track train and validation accuracy per epoch.
  • Swap in a pretrained model e.g. ResNet-18 from torchvision.models; freeze the backbone, train only a new classification head.
  • Compare the two head-to-head accuracy, training time, epochs each needed.
  • Save the better weights, write a single predict(image) function this is the bridge from notebook to app.
  • Wrap predict in a Gradio interface and launch it then upload/draw your own images and watch it work.

How you know you won. Your from-scratch CNN exceeds ~98% on MNIST test; on Pets, transfer learning clearly beats from-scratch with far less training; you can explain in plain words why a model pretrained on millions of images transfers to your 37 breeds; and the Gradio app correctly classifies an image you provide yourself.

Stretch. Add data augmentation (random flips/crops) and measure the gain. Unfreeze the top layers and fine-tune at a low learning rate. Show softmax confidence in the app. Deploy publicly with share=True or to Hugging Face Spaces and send a friend the link.

Starter code — a correct from-scratch CNN on MNIST plus a Gradio app. Predict the per-epoch accuracy trend before running, then add the transfer-learning model yourself.

import torch import torch.nn as nn import torch.nn.functional as F from torch.utils.data import DataLoader from torchvision import datasets, transforms device = "cuda" if torch.cuda.is_available() else "cpu" # 1) Data: normalize to MNIST's known mean/std. tfm = transforms.Compose([transforms.ToTensor(), transforms.Normalize((0.1307,), (0.3081,))]) train_ds = datasets.MNIST(".", train=True, download=True, transform=tfm) test_ds = datasets.MNIST(".", train=False, download=True, transform=tfm) train_dl = DataLoader(train_ds, batch_size=128, shuffle=True) test_dl = DataLoader(test_ds, batch_size=256) # 2) A small CNN from scratch. class SmallCNN(nn.Module): def __init__(self): super().__init__() self.conv1 = nn.Conv2d(1, 16, 3, padding=1) # 1 -> 16 channels self.conv2 = nn.Conv2d(16, 32, 3, padding=1) # 16 -> 32 channels self.fc1 = nn.Linear(32 * 7 * 7, 64) self.fc2 = nn.Linear(64, 10) # 10 digit classes def forward(self, x): x = F.max_pool2d(F.relu(self.conv1(x)), 2) # 28 -> 14 x = F.max_pool2d(F.relu(self.conv2(x)), 2) # 14 -> 7 x = x.flatten(1) x = F.relu(self.fc1(x)) return self.fc2(x) # raw logits model = SmallCNN().to(device) opt = torch.optim.Adam(model.parameters(), lr=1e-3) loss_fn = nn.CrossEntropyLoss() # 3) Train for a few epochs. for epoch in range(3): model.train() for xb, yb in train_dl: xb, yb = xb.to(device), yb.to(device) opt.zero_grad() loss = loss_fn(model(xb), yb) loss.backward() opt.step() # Evaluate on the test set. model.eval() correct = total = 0 with torch.no_grad(): for xb, yb in test_dl: xb, yb = xb.to(device), yb.to(device) preds = model(xb).argmax(1) correct += (preds == yb).sum().item() total += yb.size(0) print(f"epoch {epoch}: test accuracy = {correct / total:.4f}") torch.save(model.state_dict(), "mnist_cnn.pt") # 4) Deploy with Gradio. (pip install gradio) import gradio as gr import numpy as np from PIL import Image def predict(img): # In current Gradio, a Sketchpad returns a dict; the drawing is under # "composite" (an RGBA HxWx4 array). Older versions returned a plain array, # so handle both. We then convert to a single 28x28 grayscale digit. if img is None: return {} if isinstance(img, dict): img = img.get("composite") if img is None: return {} arr = np.asarray(img) # MNIST is white digits on a black background. The sketchpad strokes are # opaque, so the alpha channel (where you drew) already gives that polarity. if arr.ndim == 3 and arr.shape[2] == 4: gray = arr[:, :, 3] # alpha channel = where you drew else: gray = np.asarray(Image.fromarray(arr.astype(np.uint8)).convert("L")) pil = Image.fromarray(gray.astype(np.uint8)).resize((28, 28)) x = torch.tensor(np.array(pil), dtype=torch.float32).reshape(1, 1, 28, 28) x = (x / 255.0 - 0.1307) / 0.3081 # same normalization as training model.eval() with torch.no_grad(): probs = F.softmax(model(x.to(device)), dim=1)[0] return {str(i): float(probs[i]) for i in range(10)} demo = gr.Interface(fn=predict, inputs=gr.Sketchpad(image_mode="RGBA"), outputs=gr.Label(num_top_classes=3), title="Draw a digit") demo.launch() # add share=True for a public link # YOUR TURN: add a ResNet-18 (torchvision.models.resnet18) transfer-learning # version on Oxford-IIIT Pets and compare accuracy + training time.

After 3 epochs the from-scratch CNN should hit roughly 98–99% test accuracy on MNIST, and the Gradio window lets you draw a digit and watch the top-3 probabilities update live. The lesson lands when you redo this on Pets: from-scratch flounders, while a frozen pretrained backbone with a fresh head reaches strong accuracy in a fraction of the time.

07 · Quest 5 of 5 · the capstone · after Module 14

Quest 5 — RAG Over Your Own Docs

This is the architecture behind most real LLM products shipping today — and you're going to build it over your own documents. Ask it a question, and it answers from your files instead of making things up. Beat this and you've built something people actually pay for.

The goal. A small Retrieval-Augmented Generation (RAG) Q&A app: index your documents into a vector store, retrieve the most relevant passages for a question, and have a language model answer using only those passages — deployed via Gradio. Uses everything from Module 14 plus the embedding intuition from Modules 9 / 13.

The data. Use your own documents — a few PDFs, lecture notes, your notes from this course, a product manual, anything text. Real personal data makes this unforgettable. Embeddings via sentence-transformers, vector search via FAISS (faiss-cpu).

Your docs never use the word "energy." You ask "How do cells make energy?" Your RAG retriever — searching by embedding similarity, not keywords — returns…

Embeddings map text to vectors where meaning lives close together, regardless of exact wording. "How do cells make energy?" lands near "the mitochondrion produces ATP" even though they share no words — because the model learned they're about the same thing. That semantic matching is why RAG beats keyword search, and why it can ground an LLM in your specific documents instead of its fuzzy training memory.

LAB Quest 5 milestones

The capstone has five stages. Milestone 6 — testing that it refuses to answer what it doesn't know — is what separates a real RAG system from a confident liar.

  • Chunk your docs split into overlapping windows of a few hundred words each, so every chunk is a self-contained passage.
  • Embed every chunk and store in FAISS one vector per chunk, indexed for fast nearest-neighbor search.
  • Retrieve top-k for a question embed the question the same way, pull the most similar chunks.
  • Build the grounded prompt paste retrieved chunks as context, instruct the LLM to answer USING ONLY that context — and to say "I don't know" if it's not there.
  • Wrap it in a Gradio chat interface question in, grounded answer out.
  • Test the refusal ask things that ARE and ARE NOT in your docs; verify it won't hallucinate when it shouldn't know.

How you know you won. A question whose answer is in your docs returns a correct answer plus the source passage it used; a question outside your docs makes it say it doesn't know rather than inventing an answer; and you can explain each stage — chunk, embed, index, retrieve, generate — in one sentence apiece.

Stretch. Show the retrieved source chunks alongside each answer (citations). Compare chunk sizes and top-k values and watch the quality tradeoff. Swap the embedding model and measure retrieval quality. Add conversational memory so follow-ups work. Deploy to Hugging Face Spaces.

Starter code — the retrieval core (chunk → embed → index → retrieve). Predict which two chunks come back for the question before you run it, then add your document loader and the LLM answer step.

# pip install sentence-transformers faiss-cpu import numpy as np import faiss from sentence_transformers import SentenceTransformer # 0) YOUR TURN: replace this with text loaded from your own PDFs / notes. documents = [ "The mitochondrion is the powerhouse of the cell, producing ATP.", "Photosynthesis converts sunlight into chemical energy in plants.", "Newton's second law states that force equals mass times acceleration.", "The capital of France is Paris, located on the river Seine.", "Gradient descent updates parameters in the direction of steepest descent.", ] # 1) Chunk: here each document is already one chunk. For real docs, split long # text into ~200-word overlapping windows so each chunk stands on its own. chunks = documents # 2) Embed every chunk into a vector. embedder = SentenceTransformer("all-MiniLM-L6-v2") embeddings = embedder.encode(chunks, normalize_embeddings=True) embeddings = np.asarray(embeddings, dtype="float32") # 3) Build a FAISS index. With normalized vectors, inner product = cosine sim. index = faiss.IndexFlatIP(embeddings.shape[1]) index.add(embeddings) # 4) Retrieve the top-k most relevant chunks for a question. def retrieve(question, k=2): q = embedder.encode([question], normalize_embeddings=True).astype("float32") scores, idx = index.search(q, k) return [(chunks[i], float(scores[0][j])) for j, i in enumerate(idx[0])] question = "How do cells make energy?" for text, score in retrieve(question): print(f"{score:.3f} {text}") # YOUR TURN: # - Build a prompt that pastes these retrieved chunks as context and asks an # LLM (Hugging Face pipeline, or an API) to answer USING ONLY that context, # and to reply "I don't know" if the answer is not present. # - Wrap retrieve(...) + the LLM call in a gradio.ChatInterface.

For "How do cells make energy?" the retriever should surface the mitochondrion and photosynthesis chunks with the highest similarity — even though the question shares no exact words with either. That's the magic of embeddings: they match on meaning, not keywords. Wire a language model on top to phrase the answer from those chunks and you have a working RAG system over your own knowledge.

Why must the test set stay untouched until the very end?
tap to flip
Any decision made using the test set leaks its information into your model, so the final number stops being an honest estimate of performance on truly unseen data.
What is data leakage, and where does it sneak in during a bake-off?
tap to flip
When information from validation/test data influences training. Classic culprit: fitting a scaler or imputer on the whole dataset before splitting. Fix: put preprocessing inside a Pipeline so it re-fits per fold.
Why does transfer learning beat from-scratch training on a small image set?
tap to flip
The pretrained backbone already learned universal visual features (edges, textures, shapes) from millions of images; you only train a new head to map them to your classes. Too little data to learn those features from zero.
What does a tokenizer do, and what does the model actually receive?
tap to flip
It splits text into subword tokens and maps them to integer IDs, plus an attention mask marking real tokens vs. padding. The model never sees raw text — only those IDs.
Name the five stages of a RAG pipeline.
tap to flip
Chunk (split docs) → Embed (text to vectors) → Index (store in FAISS) → Retrieve (top-k nearest to the question) → Generate (LLM answers using only retrieved context).
How do embeddings find the right chunk when the question shares no words with it?
tap to flip
Embeddings place text with similar meaning near each other in vector space, so similarity search matches on meaning, not exact strings — "make energy" finds "mitochondria produce ATP."
